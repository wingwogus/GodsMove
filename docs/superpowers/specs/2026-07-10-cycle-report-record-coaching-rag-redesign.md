# 영농 주기 리포트·기록 피드백 RAG 재설계

- 날짜: 2026-07-10
- 상태: 승인됨
- 범위: 백엔드 도메인·애플리케이션·API 설계

## 1. 목적

영농기록 직후 제공하는 짧은 피드백과, 한 농사 주기가 끝났을 때 제공하는
상세 리포트 코칭의 책임을 분리한다.

기록 피드백은 현재 기록 한 건에 집중한다. 주기별 통계와 과거 작업 패턴은
저장형 리포트가 담당한다. 마지막 수확으로 주기가 완료되면 현재 리포트와
직전 완료 주기를 비교한 상세 코칭을 자동 생성한다.

## 2. 대체하는 기존 결정

이 문서는 다음 기존 설계의 제품 계약을 대체한다.

- `2026-07-03-medicinal-crop-record-rag-feedback-design.md`의 외부 주입형
  `TodayRecordFeedbackContext`, 주기 통계 포함 컨텍스트, 대형 공용 응답
- `2026-07-10-harvest-anchored-farming-report-design.md`의 조회 시점 실시간
  계산 방식, 지난해 완료 주기 비교, AI 코칭 비범위 결정
- `frontend/docs/Business Rule.md`의 최근 7일·30일 중심 리포트 기간 규칙
- 기존 `FarmingReportResponse`와 `GET /api/v1/reports/farming` 초안

Spring AI, pgvector, 공식문서 메타데이터, 인용 감사 등 기존 RAG 기반시설
결정은 유지한다.

## 3. 현재 문제

현재 코드에는 성격이 다른 두 RAG 경로가 공존한다.

1. `TodayRecordFeedbackService`는 풍부한 컨텍스트를 받지만 로컬 개발 API가
   전체 `TodayRecordFeedbackContext`를 요청 본문으로 받는다. 실제
   영농기록 엔티티와 연결되지 않았다.
2. `CoachingRagService`는 실제 회원·밭·작물을 조회하지만 기록 일시와 작업
   유형 정도만 프롬프트에 넣는다. 작업별 상세값을 사용하지 않는다.

추가 문제는 다음과 같다.

- 기록 피드백 컨텍스트에 최근 기록과 주기 통계가 섞여 역할이 크다.
- 하나의 `CoachingStructuredResult`가 기록 피드백과 리포트 코칭을 모두
  표현해 사용자 응답이 과도하게 크다.
- `FarmingRecordDocumentFactory`는 현재 enum 기반 작업 스키마와 맞지 않고
  실제 색인 흐름에서도 사용되지 않는다.
- 날씨 모듈은 현재 기온과 하늘 상태만 제공하며, 더미 컨텍스트에 존재하는
  7일 예보를 실제로 공급하지 않는다.
- 주기 리포트와 작업별 통계 구현이 아직 없다.

## 4. 핵심 결정

- 기록 피드백과 완료 리포트 코칭은 서로 다른 컨텍스트·출력 계약을 가진다.
- 공식 기술문서 검색, 날씨 포트, LLM 호출, 인용 감사만 공유한다.
- 사용자 영농기록은 벡터 검색으로 다시 찾지 않고 관계형 DB에서 직접 읽는다.
- 리포트 통계는 LLM이 아니라 서버의 결정론적 집계기가 계산한다.
- 진행 중 리포트와 완료 리포트를 DB에 저장한다.
- 기록 변경 시 누적값을 증감하지 않고 해당 주기의 원본을 다시 집계해
  저장된 통계를 덮어쓴다.
- 마지막 수확 저장 시 상세 리포트 코칭을 자동 생성한다.
- 완료 후 기록이 변경되면 기존 코칭은 보존하고 `STALE`로 표시한다.
- 리포트 비교 대상은 지난해가 아니라 같은 밭·작물의 직전 완료 주기다.
- 관수 횟수만으로 관수 부족·과다를 판단하지 않는다.

## 5. 단계별 범위

### 1단계: 주기 리포트 기반

- 마지막 수확 여부 저장
- 진행 중·완료 주기 판별
- 저장형 리포트와 revision
- 작업별 통계 집계
- 직전 완료 주기 비교
- 리포트 조회 API

LLM은 1단계에 포함하지 않는다.

### 2단계: 실제 기록 컨텍스트 연동

- `recordId`로 실제 영농기록과 작업별 상세 엔티티 조회
- 기록 피드백 컨텍스트 조립
- 현재 날씨·7일 예보 포트
- 피드백 생성 상태와 저장 모델
- 자동 생성·재생성 흐름

### 3단계: LLM·RAG 코칭

- 짧은 기록 피드백 프롬프트·출력·검증
- 상세 완료 리포트 코칭 프롬프트·출력·검증
- 공식문서 검색과 근거 연결
- 실패·재시도·STALE 처리

각 단계는 별도 구현 계획과 검증 게이트를 가진다.

## 6. 전체 아키텍처

```text
영농기록 DB ───────> 주기 판별·통계 집계 ───────> 저장형 주기 리포트
    │                                              │
    │                                              └─> 완료 리포트 코칭 컨텍스트
    └─> 기록 피드백 컨텍스트                              │
              │                                          │
현재 날씨·7일 예보 ─┤                                    │
공식 기술문서 RAG ──┴─> 짧은 기록 피드백      공식 기술문서 RAG ─┴─> 상세 코칭
```

권위 있는 데이터 출처는 다음과 같다.

- 영농기록·통계: PostgreSQL 원본 엔티티
- 현재 날씨·예보: 서버 날씨 포트가 반환한 데이터
- 재배·병해충 지침: 공식 기술문서 벡터 인덱스

영농기록 내용, 날씨, 계산 통계는 RAG 문서로 검색하지 않고 프롬프트의 직접
컨텍스트로 전달한다.

## 7. 영농일지 스키마 의존성

리포트 구현은 변경된 영농일지 스키마를 기준으로 한다.

- 파종/정식의 번식법과 수량 필드는 별도 개발 결과를 사용한다.
- 비료·자재는 자유 문자열이 아닌 안정적인 카테고리 코드로 저장한다.
- 약제도 자유 문자열이 아닌 안정적인 카테고리 코드로 저장한다.
- 시비 방식은 `토양시비`, `엽면시비` 2종으로 고정한다.
- 수확량은 kg 단일 단위로 저장한다.
- 약용 부위는 작물의 기본값을 복사하지 않고 수확 기록에서 선택한다.
- 수확 기록은 `isFinalHarvest: Boolean`으로 `일부 수확`과 `마지막 수확`을
  구분한다.

파종 필드의 최종 이름과 enum은 담당 개발자의 병합 결과를 따른다. 리포트
구현이 동일 필드를 중복 추가하지 않는다.

## 8. 주기 경계

주기는 회원·밭·작물 조합별로 계산한다. 같은 밭·같은 작물에 동시에 여러
주기를 진행하는 경우는 MVP에서 지원하지 않는다.

### 진행 중 주기

- 직전 마지막 수확 다음 기록부터 현재까지
- 앞선 마지막 수확이 없으면 최초 기록부터 현재까지

### 완료 주기

- 직전 마지막 수확 다음 기록부터 선택한 마지막 수확까지
- 최초 완료 주기는 최초 기록부터 선택한 마지막 수확까지
- 마지막 수확 기록은 현재 주기에 포함한다.

심기 기록은 강제 시작점이 아니라 주기 안의 기준점이다. 심기 전 밑거름과
제초 등 준비 작업도 현재 주기에 포함한다.

## 9. 저장형 리포트

### 9.1 역할

`FarmingCycleReport`는 영농기록 원본에서 다시 만들 수 있는 저장형
프로젝션이다. 원본 데이터의 대체물이 아니다.

권장 필드는 다음과 같다.

```text
id
member_id
farm_id
crop_id
status                     ACTIVE | COMPLETED
starts_at
ends_at                    nullable for ACTIVE
start_basis                FIRST_RECORD | AFTER_PREVIOUS_FINAL_HARVEST
planting_milestone_at      nullable
final_harvest_record_id    nullable for ACTIVE, required for COMPLETED
statistics_schema_version
statistics                 jsonb
source_revision
created_at
updated_at
completed_at               nullable
```

`statistics`는 타입이 있는 애플리케이션 모델을 JSONB로 직렬화한다. 임의의
문자열 맵을 서비스 전반에 전달하지 않는다.

코칭 상태는 `CoachingFeedback`이 소유한다. 리포트 행에 같은 상태를 중복
저장하지 않고, 최신 CYCLE_REPORT 피드백을 조회해 상태를 반환한다.

다음 불변식을 보장한다.

- 회원·밭·작물 조합마다 ACTIVE 리포트는 최대 하나다.
- `final_harvest_record_id`는 COMPLETED 리포트에서 유일하다.
- 마지막 수확 다음에 첫 기록이 들어오면 새 ACTIVE 리포트를 만든다.

### 9.2 직전 주기 리포트 ID를 저장하지 않는 이유

직전 주기는 같은 회원·밭·작물의 완료 리포트를 종료일 역순으로 조회해
계산한다. 과거 마지막 수확이 수정되면 직전 주기가 달라질 수 있으므로
리포트 행에 고정 FK를 저장하지 않는다.

상세 코칭 생성 당시 실제로 비교한 리포트 ID와 revision은 코칭 입력
스냅샷에 저장해 감사 가능성을 확보한다.

### 9.3 마지막 수확 기록 ID를 저장하는 이유

마지막 수확 기록 ID는 다음을 증명하는 종료 근거다.

- 사용자가 마지막 수확이라고 확인한 실제 기록
- 완료 주기의 정확한 포함 경계
- 같은 날짜에 여러 수확이 있을 때의 종료 기록
- 마지막 수확 수정·삭제 시 재계산 대상

종료 날짜만으로 이 정보를 대체할 수 없다.

## 10. 리포트 갱신

통계는 필드별 증감 연산으로 갱신하지 않는다.

```text
영농기록 생성·수정·삭제
-> 영향받는 회원·밭·작물 범위 확인
-> 해당 범위의 원본 기록 전체 조회
-> 주기 경계 재계산
-> 작업별 통계 전체 재계산
-> 저장형 리포트 덮어쓰기
-> source_revision 증가
```

- 작업일이나 상세값 변경: 해당 범위 재계산
- 밭·작물 변경: 변경 전·후 범위 모두 재계산
- 마지막 수확 여부 변경: 이후 주기 경계를 다시 나누고 관련 리포트 재계산
- 소프트 삭제: 삭제 기록을 제외하고 재계산
- 통계 계약 변경: `statistics_schema_version` 증가 후 재계산

`source_revision`은 경계 또는 저장된 통계 payload가 실제로 달라졌을 때만
증가한다. 같은 결과를 다시 계산한 것만으로 기존 코칭을 STALE로 만들지
않는다.

리포트 갱신은 외부 API와 LLM을 호출하지 않는 결정론적 DB 작업이다.
MVP에서는 영농기록 변경 트랜잭션 안에서 원본과 프로젝션을 함께 반영해
저장 직후 통계 일관성을 보장한다. LLM 생성은 같은 트랜잭션에 넣지 않는다.

동일 회원·밭·작물에 기록이 동시에 저장될 수 있으므로 ACTIVE 리포트 행을
잠근 뒤 원본을 다시 읽고 계산한다. ACTIVE 행이 아직 없는 동시 최초 생성은
유일성 제약과 재시도로 처리한다. 여러 범위를 다시 계산할 때는 ID 정렬
순서로 잠가 교착을 피한다.

## 11. 공통 통계 규칙

리포트 전체와 각 작업 유형에 다음 공통 통계를 적용한다.

| 통계 | 계산 규칙 |
| --- | --- |
| 총 기록 수 | 포함된 기록 count |
| 작업 유형별 기록 수 | `workType` group by count |
| 첫 작업일 | `min(workedAt)` |
| 최근 작업일 | `max(workedAt)` |
| 작업 기간 | 첫 작업일~최근 작업일 |
| 작업한 날짜 수 | `workedAt.toLocalDate()` distinct count |
| 평균 작업 간격 | 중복 제거한 작업 날짜를 오름차순 정렬한 인접 간격 평균 |
| 사진 첨부율 | 사진이 하나 이상인 기록 수 / 전체 기록 수 |
| 기록 당시 날씨 분포 | `weatherCondition` group by count |
| 기록 당시 평균 기온 | `avg(weatherTemperature)` |

사진 첨부율은 농사 성과가 아니라 `기록 충실도` 영역에 표시한다. 날씨
필드는 영농기록 시점의 제한된 스냅샷이므로 실제 기상 통계나 인과 판단에
사용하지 않는다.

리포트는 하나의 밭·작물로 범위가 정해지므로 농장별 횟수와 농장별 사용량을
다시 group by 하지 않는다.

### 빈 값

- 빈 수량을 0이나 평균값으로 채우지 않는다.
- 합계는 값이 있는 기록만 계산하고 `입력 건수/대상 건수`를 함께 저장한다.
- 선택형 분포는 `미입력`을 별도 항목으로 포함한다.
- 선택 필드의 평균은 만들지 않는다.
- 평균 작업 간격은 서로 다른 작업 날짜가 두 개 이상일 때만 계산한다.

### 직전 주기 비교

- 현재와 직전 완료 주기의 같은 통계 키만 비교한다.
- 지난해나 같은 날짜 구간으로 이동한 값을 비교하지 않는다.
- 직전 완료 주기가 없으면 비교값과 증감 문구를 생성하지 않는다.
- 이전 값이 0이면 백분율을 계산하지 않고 절대 차이만 표시한다.

## 12. 작업별 통계

### 12.1 파종/정식

파종 통계는 담당 개발자가 추가하는 번식법 필드를 기준으로 구현한다.

- 번식법별 작업 횟수와 비율
- 번식법별 수량 합계
- 종자량 합계
- 모종·삽수·접목묘·취목·분주 개체·조직배양 개체 수 합계

kg과 g처럼 변환 가능한 단위만 명시적인 변환 규칙으로 합산한다. 상자 같은
포장 단위와 개체 수를 임의로 합산하지 않는다.

### 12.2 관수

- 관수 기록 수
- 관수한 날짜 수
- 평균 관수 간격
- 관수량 단계 분포: 적음, 보통, 충분, 미입력
- 관수 방식별 횟수와 비율
- 직전 주기 대비 횟수·간격 변화

관수 기록은 사람이 물을 준 작업만 의미한다. 비가 와서 관수하지 않은 날과
기록을 누락한 날을 구분할 수 없으므로 다음 판단을 금지한다.

- 관수 횟수가 적으므로 수분이 부족했다.
- 직전 주기보다 관수가 적어 생육에 불리했다.
- 관수 간격이 길어 건조 피해가 예상된다.

7일 예보는 앞으로의 행동 추천에만 사용하며 과거 관수 적정성 평가에
사용하지 않는다.

### 12.3 시비

- 총 시비량 kg
- 회당 평균 시비량 kg
- 비료·자재 카테고리별 사용 횟수와 전체 대비 비율
- 비료·자재 카테고리별 사용량과 전체 대비 비율
- 토양시비·엽면시비별 횟수와 비율
- 비료·자재 카테고리별 시비 방식 교차 분포
- 직전 주기 대비 같은 카테고리의 횟수·사용량 변화

카테고리의 안정적인 ID 또는 코드를 집계 키로 사용하고 표시 이름은 별도로
저장한다.

### 12.4 병해충 방제

- 방제 기록 수와 평균 간격
- 약제 카테고리별 사용량
- 약제 단위별 사용량: ml과 g 분리
- 총 희석 살포량 L
- 대상 병해충별 횟수
- 직전 주기 대비 같은 카테고리·대상의 변화

약제 카테고리의 안정적인 ID 또는 코드를 사용한다. 여러 병해충을 한 문자열로
저장하면 정확한 빈도 계산이 어려우므로 대상 병해충도 구조화된 값 사용을
권장한다.

### 12.5 제초

- 제초 기록 수와 평균 간격
- 손제초·기계제초·멀칭·제초제별 횟수와 비율
- 미입력 횟수

수량 필드가 없으므로 제초량이나 면적당 제초량을 생성하지 않는다.

### 12.6 전정

- 공통 통계만 제공

상세 테이블이 없으므로 방식·수량 통계를 생성하지 않는다.

### 12.7 수확

- 일부·마지막 수확을 포함한 수확 횟수
- 총 수확량 kg
- 회당 평균 수확량 kg
- 첫 수확일과 마지막 수확일
- 선택한 약용 부위별 수확 횟수·수확량·비율
- 마지막 수확 기록의 생육 기간
- 부분 수확 간 생육 기간이 다르면 최소~최대 범위
- 직전 주기 대비 수확량·횟수·기간 변화

수확 단위는 kg만 허용한다.
생육 기간 단위가 월·년으로 남아 있다면 원본 표시값은 보존하고 비교 계산은
개월로 정규화한다.

## 13. 과거 강수량 후속 과제

1차 리포트에서는 관수 적정성을 평가하지 않는다. 후속 단계에서 다음 데이터를
확보한 뒤 별도 설계한다.

- 주기 전체 일별 강수량
- 강수일 수와 연속 강우일
- 가능하면 토양 수분
- 면적·토양·재배 시설 정보

과거 강수량이 추가되어도 강수와 인공 관수를 단순 합산해 수분 적정성을
단정하지 않는다.

## 14. 코칭 저장 모델

기존 `CoachingFeedback`을 생성 상태를 가진 저장 엔벨로프로 재설계한다.
기록 피드백과 리포트 코칭은 출력 JSON schema를 공유하지 않는다.

권장 공통 필드는 다음과 같다.

```text
id
member_id
feedback_type              RECORD | CYCLE_REPORT
status                     PENDING | READY | FAILED | STALE
record_id                  nullable, required for RECORD
cycle_report_id            nullable, required for CYCLE_REPORT
source_revision
input_snapshot             jsonb
structured_result          jsonb nullable until READY
citations                  jsonb
audit_status               nullable until validation
audit_warnings
failure_code               nullable
model_name                 nullable until generated
embedding_model            nullable until generated
created_at
updated_at
```

타깃 제약은 다음과 같다.

- `RECORD`: `record_id`만 존재
- `CYCLE_REPORT`: `cycle_report_id`만 존재

CHAT 실험 모드는 제품 기록 피드백·리포트 코칭과 분리한다.

동일한 원본 버전의 중복 생성을 막기 위해
`feedback_type + target_id + source_revision`을 멱등성 키로 사용한다.
PENDING 행은 원본 변경 트랜잭션에서 함께 저장하고 LLM 실행은 커밋 이후에
시작한다. 실행 중 프로세스가 종료되어도 남아 있는 PENDING 행을 재처리할 수
있어야 한다.

`source_revision`의 의미는 타입별로 다르다.

- RECORD: FarmingRecord, 작업 상세, 연결 미디어 ID의 변경 버전
- CYCLE_REPORT: FarmingCycleReport의 `source_revision`

현재 날씨와 예보는 생성 시점의 `input_snapshot`에 보존하지만, 예보가 시간이
지나 바뀌었다는 이유만으로 기존 기록 피드백을 STALE로 바꾸지 않는다.

## 15. 코칭 상태

```text
PENDING -> READY
        -> FAILED
READY   -> STALE
STALE   -> PENDING -> READY
```

- 기록 저장과 통계 리포트 갱신은 LLM 실패로 롤백하지 않는다.
- 마지막 수확 저장 직후 완료 리포트 코칭 `PENDING`을 자동 생성한다.
- 완료 기록 변경으로 `source_revision`이 달라지면 기존 코칭을 `STALE`로
  바꾼다.
- STALE 코칭은 보존하고 사용자가 재생성을 요청할 때 새 결과를 만든다.
- 기록 피드백은 기록 생성·수정 후 자동 생성한다. 수정 전 결과는 STALE로
  보존한다.

## 16. 기록 피드백 컨텍스트

입력은 외부 JSON이 아니라 인증 회원 ID와 `recordId`다.

`RecordFeedbackContextAssembler`는 다음을 직접 조회한다.

- 회원의 영농 경험·경영 형태
- 밭 이름·주소·위경도
- 작물 이름·약용 부위 기본 정보
- 대상 `FarmingRecord`
- 작업 유형별 상세 엔티티
- 기록 당시 `weatherCondition`, `weatherTemperature`
- 메모와 사진 첨부 수
- 현재 날씨와 7일 예보

다음은 기록 피드백 컨텍스트에서 제외한다.

- 같은 주기의 이전 기록
- 주기별 작업 횟수
- 최근 30일 작업 횟수
- 직전 주기 통계
- 파종 후 경과일

현재 도메인의 `WorkType`과 상세 enum을 직접 매핑하고 별도
`TodayRecordWorkType` enum을 중복 유지하지 않는다.

## 17. 기록 피드백 출력

```json
{
  "goodPoint": {
    "basis": "점적관수 방식과 토양 상태 기록",
    "text": "토양 상태를 보고 관수한 점이 좋아요",
    "evidenceRefs": ["record:..."]
  },
  "nextActions": [
    {
      "due": "NEXT_WEEK",
      "category": "WEATHER",
      "basis": "다음 주 비 예보",
      "text": "다음 주 비 예보가 있으니 배수로를 확인하세요",
      "evidenceRefs": ["weather:..."]
    }
  ]
}
```

### 출력 규칙

- 잘한 점 정확히 1개
- 다음 행동 2~3개
- 잘한 점과 다음 행동 모두 텍스트 안에 판단 근거를 명시
- 각 항목에 `basis`, `text`, `evidenceRefs` 필수
- 문구 목표는 약 25자, 허용 범위는 15~45자
- 문장을 강제로 자르지 않고 길이 실패 시 한 번 재생성
- 날씨·병해충 행동은 관련 근거가 있을 때만 포함
- 근거 없는 일반적 권장 금지

`basis`는 감사와 근거 보기에 사용하는 구조화 필드이고, `text`는 농부가
바로 읽는 자연스러운 한 문장이다.

기존 `riskLevel`, 긴 diagnosis, observations, follow-up questions,
record quality score는 기록 피드백 사용자 응답에서 제거한다. 신뢰도, 감사
결과, 모델 정보는 운영 데이터로만 저장한다.

## 18. 완료 리포트 코칭 컨텍스트

`CycleReportCoachingContextAssembler`는 다음을 사용한다.

- 완료 리포트 ID와 revision
- 현재 주기 공통·작업별 통계
- 직전 완료 주기 ID, revision, 같은 통계
- 현재와 직전 주기의 계산된 차이
- 밭과 작물 정보
- 관련 공식 기술문서 근거

생성 당시 비교 리포트와 두 revision, LLM 입력 통계는
`input_snapshot`에 저장한다.

## 19. 완료 리포트 코칭 출력

```json
{
  "overallReview": "이번 주기는 시비 간격이 일정했고 수확량도 직전 주기보다 증가했습니다.",
  "strengths": [
    {
      "basis": "시비 간격 12일, 직전 주기 18일",
      "text": "이번 주기는 시비 간격을 더 일정하게 유지했어요.",
      "evidenceRefs": ["report:current:..."]
    }
  ],
  "improvements": [],
  "nextCycleActions": [],
  "cautions": []
}
```

- 종합 평가 2~4문장
- 잘한 운영 패턴 2~3개
- 개선이 필요한 패턴 1~3개
- 다음 주기 행동 계획 3~5개
- 주의사항 0~3개
- 각 항목에 현재 통계, 직전 통계 또는 공식문서 근거 연결

`strengths`, `improvements`, `nextCycleActions`, `cautions`의 각 항목은
`basis`, `text`, `evidenceRefs`를 가진다. 다음 주기 행동은 추가로 우선순위를
가진다.

화면의 원본 통계와 비교 표를 그대로 반복하지 않고 의미 있는 변화만 골라
해석한다.

### 금지 규칙

- 강수 근거 없이 관수 부족·과다 판단
- 통계 차이를 원인·결과로 단정
- 서로 다른 단위·카테고리 합산
- 직전 완료 주기가 없는데 증가·감소 표현
- 공식문서 근거 없는 비료량·약제량 처방
- 입력되지 않은 생육 상태·병해충 추측

## 20. 근거 모델

근거 타입은 다음과 같다.

- `RECORD`: 현재 영농기록과 상세값
- `WEATHER`: 현재 날씨 또는 7일 예보
- `REPORT`: 저장된 리포트 통계와 revision
- `DOCUMENT`: 공식 기술문서 청크

서버가 허용 근거 ID 목록을 만들고 LLM은 그 목록 안의 값만 참조한다.
사용자에게는 자연어 문장을 먼저 보여주고 필요할 때 basis와 출처를 확인할
수 있게 한다.

기록 피드백 검증기는 정규화한 `basis`의 핵심 내용어가 `text`에도 포함되는지
확인한다. 단순한 citation 존재만으로 근거가 문장에 드러났다고 인정하지 않는다.

## 21. 검색 규칙

벡터 검색 대상은 공식 기술문서다.

기록 피드백 검색 쿼리는 다음 값에서 만든다.

- 작물
- 작업 유형
- 작업별 카테고리·상세값
- 메모의 병징·작업 목적
- 실제 날씨·예보 위험 신호

완료 리포트 검색 쿼리는 다음 값에서 만든다.

- 작물
- 통계 변화가 큰 작업 유형
- 기록된 병해충 카테고리
- 다음 주기 계획에 필요한 공식 재배 지침

사용자 기록과 저장된 리포트를 벡터스토어에 다시 색인하지 않는다.

## 22. 생성 흐름

### 기록 피드백

```text
기록 생성·수정 완료
-> 기존 READY 결과가 있으면 STALE
-> RECORD/PENDING 생성
-> 실제 엔티티 컨텍스트 조립
-> 공식문서 검색
-> LLM 생성·검증
-> READY 또는 FAILED
```

### 완료 리포트 코칭

```text
마지막 수확 저장
-> 리포트 최종 재계산
-> ACTIVE를 COMPLETED로 변경
-> 직전 완료 주기 조회
-> CYCLE_REPORT/PENDING 자동 생성
-> 상세 컨텍스트 조립
-> 공식문서 검색
-> LLM 생성·검증
-> READY 또는 FAILED
```

### 완료 후 기록 변경

```text
원본 기록 변경
-> 리포트 전체 재계산
-> source_revision 증가
-> 기존 상세 코칭 STALE
-> 기존 코칭 보존
-> 사용자 재생성 요청 시 새 PENDING 생성
```

## 23. 실패 처리

- 날씨 조회 실패: 날씨 행동을 제외하고 기록·문서 근거로 생성
- 공식문서 부족: 비료량·약제량·병해충 처방을 생성하지 않음
- READY 계약의 행동 2개를 근거 있게 만들 수 없음:
  `FAILED/INSUFFICIENT_EVIDENCE`
- 허용하지 않은 근거 ID: 출력 검증 실패
- 구조화 출력 또는 길이 실패: 한 번 재생성 후 `FAILED`
- LLM·검색 실패: 영농기록과 리포트 저장은 유지
- 직전 주기 없음: 현재 주기 단독 코칭, 비교 문구 없음

## 24. API 방향

### 리포트

```text
GET  /api/v1/farming-reports/current?farmId={farmId}&cropId={cropId}
GET  /api/v1/farming-reports?farmId={farmId}&cropId={cropId}&status={status}
GET  /api/v1/farming-reports/{reportId}
GET  /api/v1/farming-reports/{reportId}/coaching
POST /api/v1/farming-reports/{reportId}/coaching/regenerate
```

목록은 기존 커서 페이지 규칙을 따른다. 재생성은 COMPLETED 리포트의
STALE 또는 FAILED 코칭에만 허용한다.

### 기록 피드백

```text
GET  /api/v1/farming-records/{recordId}/coaching-feedback
POST /api/v1/farming-records/{recordId}/coaching-feedback/regenerate
```

조회는 현재 원본 revision과 일치하는 가장 최신 생성 시도를 반환한다. 현재
revision의 결과가 없으면 가장 최신 STALE 결과를 반환한다. 재생성은 STALE
또는 FAILED에 허용한다.

프로덕션 API는 `TodayRecordFeedbackContext` 전체를 요청 본문으로 받지 않는다.
로컬 실험 도구가 필요하면 제품 컨트롤러와 분리된 local profile에서만
유지한다.

## 25. 검증 전략

### 주기·프로젝션

- 최초 기록부터 진행 중 리포트 생성
- 이전 마지막 수확 다음 기록부터 새 주기 생성
- 일부 수확은 현재 주기를 유지
- 마지막 수확은 현재 주기를 완료
- 심기 전 준비 작업 포함
- 생성·수정·삭제 후 전체 재계산
- 밭·작물 변경 시 전후 범위 재계산
- 마지막 수확 변경 시 주기 분할·병합 재계산
- revision 증가와 코칭 STALE 전이

### 통계

- 동일 작업일 중복을 제거한 평균 간격
- 빈 값 coverage와 미입력 분포
- 시비·약제 카테고리 코드 집계
- ml과 g 분리
- 수확 kg 합계
- 약용 부위별 분포
- 마지막 수확 생육 기간과 부분 수확 범위
- 직전 완료 주기 비교와 0 분모 처리
- 강수 근거 없는 관수 적정성 해석 차단

### 실제 기록 컨텍스트

- 모든 `WorkType`의 상세 엔티티 매핑
- 회원 소유권 확인
- 사진은 첨부 수만 전달하고 이미지 분석하지 않음
- 이전 기록·주기 통계·최근 30일 통계 제외
- 날씨 실패 시 축소 컨텍스트 생성

### 기록 피드백

- 잘한 점 1개
- 다음 행동 2~3개
- basis·text·근거 ID 검증
- 텍스트 안에 근거와 행동 포함
- 길이 범위 검증과 한 번 재생성
- 병해충·날씨 근거가 없을 때 관련 행동 미생성
- 허용 근거 외 ID 차단

### 완료 리포트 코칭

- 마지막 수확 후 자동 PENDING 생성
- 현재·직전 리포트 revision 스냅샷
- 직전 주기 없음 처리
- 완료 기록 수정 후 STALE
- 수동 재생성 후 새 READY 결과
- 통계를 인과 관계로 단정하지 않는 출력 검증

### API

- 다른 회원의 기록·리포트·코칭 접근 차단
- PENDING·READY·FAILED·STALE 응답
- LLM 실패 시 영농기록과 리포트 유지
- 더미 컨텍스트 제품 API 제거

## 26. 비범위와 후속 과제

- 과거 강수량과 토양 수분을 이용한 관수 적정성 평가
- 사진 내용 분석과 병징 이미지 판독
- 같은 밭·작물의 중첩 주기
- 자동 비료량·농약량 처방
- 의료 효능·복용 조언
- 진행 중 주기의 LLM 종합 코칭

## 27. 완료 기준

- 진행 중 리포트가 기록 변경 직후 저장된 통계로 갱신된다.
- 마지막 수확이 완료 리포트를 만들고 상세 코칭을 자동 요청한다.
- 완료 리포트는 직전 완료 주기와 비교한다.
- 기록 피드백은 실제 엔티티·날씨·공식문서만 사용한다.
- 기록 피드백은 근거가 문장에 포함된 잘한 점 1개와 행동 2~3개를 반환한다.
- 완료 리포트 코칭은 현재·직전 통계와 공식문서를 근거로 상세하게 생성된다.
- 원본 변경 시 통계 revision이 증가하고 기존 상세 코칭은 STALE이 된다.
- LLM 실패가 영농기록과 리포트 저장을 무효화하지 않는다.
