# 리포트 피드백 역할별 최소 길이 설계

## 목적

리포트 코칭이 지나치게 짧아 근거와 행동 맥락이 빠지는 문제를 줄인다. 기존 최대
50자 계약 안에서 섹션 역할에 따라 서로 다른 최소 길이를 적용한다.

## 길이 계약

- `summary`: 20~50자
- `comparisons[].text`: 25~50자
- `strengths[].text`: 30~50자
- `improvements[].text`: 30~50자
- `nextActions[].text`: 30~50자

정확히 최소 길이와 50자는 허용한다. `basis`와 `evidenceRefs`는 내부 근거 필드이므로
길이를 제한하지 않는다. 기록 피드백의 15~60자 검증 계약은 변경하지 않는다.

## 생성과 검증

`ReportFeedbackPromptBuilder`는 역할별 허용 범위를 명시하고, 짧은 문장을 의미 없는
표현으로 채우지 말고 근거·판단·실행 방법 중 해당 섹션에 필요한 내용을 보강해 다시
쓰도록 지시한다.

`ReportFeedbackOutputValidator`는 기존 최대 50자 검사에 역할별 최소값을 결합한다.
범위를 벗어난 `summary`는 기존 `summary_text_length`, 각 섹션은 기존
`<section>_text_length` 경고를 사용한다. 새 진단 코드를 만들지 않으므로
`ReportFeedbackGenerationService`와 `ReportFeedbackGenerationHandler`의 안전
allowlist는 변경하지 않는다.

길이 미달 결과는 기존 구조화 출력 흐름에서 한 번 재생성한다. 생성 문구를 강제로
자르거나 뒤에 상투적인 표현을 붙이지 않으며, 두 번째 결과도 범위를 벗어나면 기존
`STRUCTURED_OUTPUT_INVALID` 실패로 처리한다.

## 제외 사항

- API 응답 구조와 데이터베이스 스키마를 변경하지 않는다.
- 이미 저장된 리포트 피드백을 수정하지 않는다.
- 최소 길이를 맞추기 위한 서버 측 문자열 패딩을 추가하지 않는다.
- 문단 개수, 줄바꿈 금지, 근거 참조 검증을 변경하지 않는다.

## 검증

- 프롬프트가 다섯 출력 위치의 역할별 길이 범위를 명시하는지 확인한다.
- `summary` 20자, 비교 25자, 핵심 코칭 30자와 모든 항목 50자를 허용한다.
- `summary` 19자, 비교 24자, 핵심 코칭 29자를 각각 기존 길이 진단으로 거부한다.
- 길이 미달 진단이 원문 없이 재시도 프롬프트로 전달되는지 확인한다.
- 기존 테스트 fixture를 새 계약 안의 자연스러운 문장으로 조정한다.
- 관련 집중 테스트, API 계약 테스트, 캐시를 무시한 전체 백엔드 테스트를 실행한다.

## 완료 기준

- 새로 저장되는 리포트 피드백은 역할별 최소 길이와 최대 50자를 모두 만족한다.
- 짧은 생성 결과는 저장되지 않고 기존 한 번의 안전 재생성으로 복구된다.
- 기존 API·DB·근거 검증과 기록 피드백 길이 계약에 회귀가 없다.
