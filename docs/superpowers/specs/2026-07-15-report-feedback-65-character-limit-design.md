# 리포트 피드백 65자 상한 설계

## 목적

리포트 코칭이 근거와 실행 맥락을 자연스럽게 담을 수 있도록 기존 50자 상한을
65자로 늘린다. 방금 적용한 역할별 최소 길이는 그대로 유지한다.

## 길이 계약

- `summary`: 20~65자
- `comparisons[].text`: 25~65자
- `strengths[].text`: 30~65자
- `improvements[].text`: 30~65자
- `nextActions[].text`: 30~65자

정확히 65자는 허용하고 66자부터 거부한다. `basis`와 `evidenceRefs`는 제한하지
않으며 기록 피드백의 15~60자 계약은 변경하지 않는다.

## 생성과 검증

`ReportFeedbackPromptBuilder`의 역할별 범위 상한을 모두 65자로 변경한다. 최대
길이를 맞출 때 문장을 강제로 자르지 않고 핵심 내용을 남겨 다시 쓰게 한다.

`ReportFeedbackOutputValidator`의 공통 `MAX_TEXT_LENGTH`를 65로 변경한다. 최소값,
빈 문구, 한 문단, 섹션 개수, 근거 참조 검증은 그대로 유지한다. 범위를 벗어난
결과는 기존 `summary_text_length`와 `<section>_text_length` 진단을 사용하므로 안전
재시도와 로그 allowlist는 변경하지 않는다.

## 제외 사항

- API 응답 구조와 데이터베이스 스키마를 변경하지 않는다.
- 이미 저장된 리포트 피드백을 수정하지 않는다.
- 문자열을 65자에서 강제로 자르거나 최소 길이까지 채우지 않는다.
- 기록 피드백의 길이 프롬프트와 검증 범위를 변경하지 않는다.

## 검증

- 프롬프트가 20~65자, 25~65자, 30~65자 범위를 명시하는지 확인한다.
- `summary`와 모든 섹션 `text`가 정확히 65자일 때 허용되는지 확인한다.
- 66자인 `summary`와 모든 섹션 `text`가 기존 길이 진단으로 거부되는지 확인한다.
- 20자, 25자, 30자의 기존 최소 경계가 계속 허용되는지 확인한다.
- 관련 집중 테스트, API 계약 테스트, 캐시를 무시한 전체 백엔드 테스트를 실행한다.

## 완료 기준

- 새로 저장되는 리포트 피드백은 역할별 최소값 이상, 최대 65자 이하이다.
- 66자 이상 출력은 저장되지 않고 기존 한 번의 안전 재생성으로 처리된다.
- API·DB·근거 검증과 기록 피드백 길이 계약에 회귀가 없다.
