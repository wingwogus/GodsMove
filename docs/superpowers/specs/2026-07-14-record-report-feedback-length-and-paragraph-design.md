# 기록·리포트 피드백 문구 길이와 문단 계약 설계

## 배경

기록 피드백은 현재 `goodPoint.text`와 `nextActions[].text`에 같은 생성 길이
목표를 적용한다. 두 문구는 화면에서 맡는 역할이 다르므로, 잘한 점은 더 짧게
유도하고 다음 행동은 기존 길이를 유지한다.

리포트 피드백은 `comparisons`, `strengths`, `improvements`, `nextActions`를
배열로 제공하지만 항목 수를 제한하지 않는다. 앞으로 각 화면 섹션은 목록이
아니라 하나의 자연스러운 문단을 보여 준다. 기존 API 호환성을 위해 배열 형태와
`summary`는 유지한다.

## 목표

- 기록 피드백의 잘한 점은 15~23자, 다음 행동은 15~25자를 생성 목표로 삼는다.
- 기록 피드백의 60자 검증 안전망은 유지한다.
- 리포트 피드백의 잘한 점, 개선 필요점, 추천 행동은 각각 정확히 한 문단을
  생성한다.
- 직전 리포트 비교는 비교 데이터가 있을 때 정확히 한 문단, 없을 때 빈 배열로
  생성한다.
- 기존 리포트 피드백 API 필드와 배열 형태를 유지한다.

## 비목표

- 기록 피드백 API 또는 저장 스키마를 변경하지 않는다.
- 리포트 피드백 배열을 단일 문자열 필드로 바꾸지 않는다.
- 리포트 피드백 `summary`를 제거하거나 의미를 바꾸지 않는다.
- 화면 제목이나 섹션 제목을 변경하지 않는다. `참참참의 코칭`, `잘한 점`,
  `이전 리포트와의 비교`, `개선 필요점`, `추천 행동`은 표시 계층의 책임이다.
- 문장을 강제로 잘라 길이를 맞추지 않는다.

## 기존 동작

### 기록 피드백

- 프롬프트는 모든 `text`에 15~25자 목표를 함께 적용한다.
- 출력 검증기는 모든 `text`에 15~60자 범위를 적용한다.
- 과거에 25자를 검증 상한으로 강제했을 때 정상적인 모델 응답도 폐기되어,
  검증 상한을 60자로 복구한 이력이 있다.

### 리포트 피드백

- `summary`는 필수 문자열이다.
- `comparisons`, `strengths`, `improvements`, `nextActions`는 배열이다.
- 비교 데이터가 없을 때 `comparisons`가 비어 있어야 한다.
- 나머지 배열은 근거가 없을 때 비어 있어도 되며, 항목 수 상한이 없다.
- 생성 결과가 출력 검증에 실패하면 기존 생성 흐름이 한 번 재시도한다.

## 설계

### 1. 기록 피드백 생성 목표 분리

`RecordFeedbackPromptBuilder`의 공통 길이 지시를 역할별 지시로 분리한다.

- `goodPoint.text`: 15~23자를 목표로 작성한다.
- `nextActions[].text`: 각각 15~25자를 목표로 작성한다.
- 두 역할 모두 강제로 자르지 않고 문장을 다시 써서 목표를 맞춘다.

`RecordFeedbackOutputValidator`의 15~60자 검증 범위는 변경하지 않는다. 23자와
25자는 모델 출력을 유도하는 제품 문구 목표이며, 짧은 초과만으로 전체 피드백을
실패시키는 하드 제한이 아니다.

### 2. 리포트 피드백 섹션별 한 문단

`ReportFeedbackContent`와 공개 API의 배열 형태를 유지하면서 생성 개수를 다음과
같이 제한한다.

| 필드 | 생성 개수 | 표시 의미 |
| --- | ---: | --- |
| `comparisons` | 비교 데이터가 있으면 1개, 없으면 0개 | 이전 리포트와의 비교 |
| `strengths` | 1개 | 잘한 점 |
| `improvements` | 1개 | 개선 필요점 |
| `nextActions` | 1개 | 추천 행동 |

각 배열의 단일 항목 `text`가 한 문단을 이룬다. 한 문단에는 자연스럽게 이어지는
여러 문장이 들어갈 수 있지만 `\r` 또는 `\n` 줄바꿈은 포함하지 않는다. 프롬프트는
목록 기호나 항목 나열 대신 한 문단으로 서술하도록 지시한다.

`summary`는 현재 계약대로 유지한다. 섹션 문단은 서로 역할이 겹치지 않아야 한다.

- `comparisons`: 서버가 제공한 현재·직전 동일 작업 비교 사실만 설명한다.
- `strengths`: 현재 리포트와 기록에서 확인되는 좋은 관리 판단을 회고한다.
- `improvements`: 현재 리포트, 기록 또는 공식문서 근거로 개선할 점을 설명한다.
- `nextActions`: 근거에 기반해 다음에 실행할 구체적인 행동을 제안한다.

공식문서가 없더라도 `strengths`, `improvements`, `nextActions`는 현재 리포트와
대상 기록을 근거로 각각 한 문단을 생성한다. 공식문서가 필요한 기술적 주장은
문서 근거 없이 만들지 않는다.

### 3. 출력 검증

`ReportFeedbackOutputValidator`에 다음 구조 검증을 추가한다.

- `strengths.size == 1`
- `improvements.size == 1`
- `nextActions.size == 1`
- 서버 비교 데이터가 있으면 `comparisons.size == 1`
- 서버 비교 데이터가 없으면 `comparisons.isEmpty()`
- 각 섹션 항목의 `text`에 줄바꿈이 없음

개수 또는 문단 형식 위반은 원문을 포함하지 않는 안정적인 검증 코드로 반환한다.
기존의 빈 값, 허용 근거 ID, 비교 리포트 근거 검증은 그대로 적용한다.

### 4. 데이터 흐름과 실패 처리

1. 서버가 현재 리포트, 동일 작업 기록, 직전 리포트 비교값과 공식문서를 준비한다.
2. 프롬프트가 섹션별 역할과 정확한 항목 수를 모델에 지시한다.
3. 구조화된 `ReportFeedbackContent`를 역직렬화한다.
4. 출력 검증기가 항목 수, 한 문단 형식, 근거 무결성을 검증한다.
5. 실패하면 기존 정책대로 한 번 재생성한다.
6. 두 번째 결과도 유효하지 않으면 기존 실패 상태와 안전한 오류 코드를 사용한다.
7. 성공 결과는 기존 엔티티와 API 배열 계약으로 저장·반환한다.

새로운 예외 유형, 재시도 횟수, 저장 스키마는 추가하지 않는다.

## API 호환성

공개 응답 형태는 바뀌지 않는다.

```json
{
  "summary": "이번 물 주기 흐름을 함께 돌아봤어요.",
  "comparisons": [
    { "text": "이전보다 작업을 이틀 빨리 시작했으며 방법은 꾸준히 유지했어요." }
  ],
  "strengths": [
    { "text": "마른 부분부터 살피고 축축한 곳에서는 멈춰 상황에 맞게 조절했어요." }
  ],
  "improvements": [
    { "text": "두 기록 모두 물의 양이 적어 흙 속까지 충분히 젖었는지 확인할 필요가 있어요." }
  ],
  "nextActions": [
    { "text": "다음에는 물을 준 뒤 손가락으로 흙 속의 젖은 깊이를 확인하세요." }
  ]
}
```

직전 비교 데이터가 없으면 `comparisons`만 빈 배열이며 나머지 세 섹션은 각각
한 항목을 유지한다.

## 테스트

### 기록 피드백

- 프롬프트가 잘한 점의 15~23자 목표를 포함한다.
- 프롬프트가 다음 행동의 15~25자 목표를 포함한다.
- 출력 검증기의 15~60자 안전망이 유지된다.

### 리포트 피드백

- 프롬프트가 섹션별 정확한 항목 수와 한 문단 규칙을 포함한다.
- `strengths`, `improvements`, `nextActions`가 각각 한 개면 통과한다.
- 세 필수 섹션 중 하나가 비거나 두 개 이상이면 해당 개수 오류로 거절한다.
- 비교 데이터가 있으면 `comparisons` 한 개만 허용한다.
- 비교 데이터가 없으면 빈 `comparisons`만 허용한다.
- 섹션 `text`에 줄바꿈이 있으면 문단 형식 오류로 거절한다.
- 기존 근거 ID와 비교 리포트 근거 검증이 계속 동작한다.
- 컨트롤러 응답의 `summary`와 네 배열 필드가 유지된다.

관련 애플리케이션 테스트와 API 테스트를 실행해 회귀를 확인한다.

## 위험과 완화

- 필수 섹션을 정확히 한 개로 강제하면 모델 생성 실패율이 높아질 수 있다.
  프롬프트에 개수와 역할을 명확히 쓰고 기존 한 번의 재시도를 유지한다.
- 한 항목 안의 문단이 지나치게 길어질 수 있다. 이번 변경은 항목 수와 문단 형태만
  다루며, 리포트 문단 길이 제한은 실제 생성 결과를 본 뒤 별도 요구로 결정한다.
- 현재 작업 트리에 리포트 피드백 관련 미커밋 변경이 있다. 구현은 해당 변경을
  보존하고 길이·항목 수·문단 규칙에 필요한 최소 줄만 수정한다.

## 완료 조건

- 기록 피드백 프롬프트가 잘한 점 23자와 다음 행동 25자 목표를 구분한다.
- 기록 피드백 검증 상한은 60자로 유지된다.
- 리포트 피드백 필수 세 섹션은 각각 한 항목만 허용한다.
- 비교 섹션은 비교 데이터 유무에 따라 정확히 1개 또는 0개다.
- 모든 리포트 섹션 항목은 줄바꿈 없는 한 문단이다.
- 기존 리포트 피드백 API 응답 형태와 저장 스키마가 유지된다.
- 관련 테스트가 통과한다.
