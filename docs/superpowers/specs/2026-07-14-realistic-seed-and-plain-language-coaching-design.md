# 현실적인 귀농 시드와 쉬운 코칭 문체 설계

## 배경과 문제

로컬 RAG 시드는 현재 수확 전 단일 주기의 기록 여섯 건만 만든다. 회원과 농장 이름에
`로컬`, `테스트`가 들어가고 모든 메모가 보고서처럼 정돈된 존댓말이라 실제 농부의
현장 기록처럼 보이지 않는다. 직전 완료 주기도 없으므로 리포트 코칭은 작업별 이전
주기 비교를 검증할 수 없다.

코칭 문체에도 별도 문제가 있다. 기록 코칭 프롬프트는 쉬운 말을 요구하지만 모델
입력에 `DRIP`, `NORMAL`, `NON_REGISTERED_FARMER` 같은 enum 값과 `관수`, `생육`,
`배수 상태` 같은 용어를 함께 제공한다. 리포트 코칭은 `WATERING`, 영어 통계 필드와
상세 맵까지 그대로 문자열로 전달한다. 두 출력 검증기는 공백, 존댓말, 근거 ID,
중복 등을 확인하지만 영어, raw enum, 어려운 농업 용어는 거부하지 않는다. 따라서
프롬프트 지시와 실제 통과 조건이 일치하지 않는다.

이번 변경은 다음 두 목표를 하나의 로컬 검증 시나리오로 묶는다.

1. 귀농 2년 차 농부의 이전 완료 주기와 현재 수확 전 주기를 현실적인 현장 메모로
   제공한다.
2. 기록 코칭과 작업별 리포트 코칭 모두 사용자에게 보여 주는 문장에서 영어,
   raw enum, 확인된 어려운 농업 용어를 차단한다.

## 확정된 결정

- 페르소나는 강원특별자치도 평창군 진부면에서 참당귀를 재배하는 귀농 2년 차
  농부다.
- 이전 주기는 2024년 봄부터 가을까지이며 최종 수확으로 완료한다.
- 현재 주기는 2025년 봄부터 수확 직전까지만 시드한다.
- 현재 주기의 최종 수확은 HTML에서 사용자가 직접 등록한다.
- 날짜는 실행일에 따라 계절이 밀리지 않도록 고정한다.
- 이전 주기 여덟 건, 현재 주기 일곱 건만 둔다. 긴 일기 형태의 과도한 기록은
  추가하지 않는다.
- 두 주기에 심기, 물 주기, 거름 주기, 병 관리, 풀 뽑기를 공통으로 넣고 현재
  최종 수확까지 등록되면 수확도 공통 작업이 된다.
- 이전 주기는 시행착오를, 현재 주기는 관찰 후 조절한 행동을 담아 기록 목록 자체가
  자연스러운 두 해의 농사 이야기로 보이게 한다.
- 현재 ReportFeedback은 직전 리포트의 동일 작업 통계만 비교한다. 이전 메모까지
  비교하는 것처럼 표현하지 않으며, 물 준 양·거름 양·약 사용량·수확량처럼 현재
  계약이 실제 전달하는 통계 차이를 비교 근거로 사용한다.
- 이전 완료 리포트는 서버 시작 시 먼저 투영한다. 이전 주기 작업별 코칭을 먼저
  시작해 현재 최종 수확 시 모든 과거·현재 코칭이 한꺼번에 생성되는 일을 줄인다.
- 기록 코칭과 리포트 코칭은 동일한 쉬운 말 정책을 사용한다.
- 7일보다 오래된 기록 코칭에는 현재 날씨와 향후 예보를 붙이지 않는다. 기록 당시
  날씨만 사용해 과거 기록에 오늘의 날씨가 섞이지 않게 한다.
- 검색 질의와 공식 기술 문서는 검색 정확도 때문에 전문용어를 유지한다.
- 도메인 enum과 API wire 값은 바꾸지 않는다. 사용자 문장과 HTML 표시만 쉽게
  바꾼다.
- 생성 후 문자열을 치환하지 않는다. 의미가 달라질 수 있으므로 검증 실패 시 기존
  구조화 출력 재시도를 사용한다.
- 공통 생성 서비스, 범용 프롬프트 계층, 거대한 농업 용어 사전은 만들지 않는다.
- local 프로필 서버는 기본적으로 loopback에만 바인딩한다. 물리 기기 테스트처럼
  LAN 노출이 필요한 개발자는 환경 변수로 명시적으로 해제한다.

## 변경 범위

### Git으로 관리하는 운영 코드

- `coaching/common`의 작은 사용자 문장 정책
- 기록 코칭 프롬프트 입력 표현과 출력 검증
- 리포트 코칭 프롬프트 입력 표현과 출력 검증
- 과거 기록에 live weather를 붙이지 않는 기록 코칭 컨텍스트 조립 규칙
- 두 생성 서비스의 안전한 재시도 오류 코드
- local 프로필의 기본 loopback 바인딩
- 관련 application/api 회귀 테스트

### 기존 정책대로 로컬에만 두는 코드

- `DevRagSeedService`와 테스트
- local profile의 시드 실행기와 테스트
- `frontend/dev-rag-test.html`

위 로컬 파일은 `.gitignore` 정책을 유지하며 강제로 Git에 추가하지 않는다. 로컬
로그인용 고정 식별자와 자격 증명은 기존 값을 유지하되 추적되는 문서나 커밋 본문에
복사하지 않는다.

### 이번 범위가 아닌 것

- 도메인 enum 이름이나 전역 `label` 변경
- API 응답의 `workType`, `due`, `category`, `status` wire 값 변경
- 이미 `READY`인 코칭의 자동 재생성 또는 마이그레이션
- 현재 ReportFeedback 컨텍스트에 이전 주기 기록·메모 추가
- 외부 농업 용어 사전, 형태소 분석기, 두 번째 LLM 감사 호출
- 새로운 메시지 큐, 전용 비동기 실행기, 재시도 횟수 증가
- 운영 데이터 마이그레이션

기준이 없으면 모델이 정답 수치를 만들어 내지 않는 기존 규칙을 유지한다.

### 시드 구조

`DevRagSeedService` 안에서 기존 명시적 저장 방식을 유지한다.

```text
seedReportFeedbackScenario
  -> 회원/농장/회원작물 생성
  -> seedPreviousCompletedCycle
  -> seedCurrentCycleBeforeHarvest
  -> FarmingCycleReportProjectionService.rebuild
```

별도 fixture DSL, 빌더 계층, 시드 버전 테이블은 만들지 않는다. 작업별 private helper는
기존 상세 엔티티 저장 책임을 재사용한다. 완료 주기를 위해 `HarvestRecordRepository`를
추가하고 이전 최종 수확 상세를 명시적으로 저장한다.

기존 이메일 기반 멱등성 가드는 유지한다. 로컬 프로필은 일반적으로
`ddl-auto: create`이므로 서버 재시작 시 새 시나리오가 적용된다. 스키마를 유지하는
설정에서는 기존 시드 회원이 있으면 전체 시드가 건너뛰어지므로 검증 전에 로컬 DB를
재생성해야 한다.

### 서버 시작 시 이전 리포트 선투영

모든 기록을 직접 저장한 뒤 시드 트랜잭션 안에서 기존
`FarmingCycleReportProjectionService.rebuild`를 한 번 호출한다. 파티셔너는 이전
최종 수확을 경계로 다음을 만든다.

```text
2024 기록 + 최종 수확 -> COMPLETED 이전 리포트
2025 수확 전 기록     -> ACTIVE 현재 리포트
```

완료된 이전 리포트만 작업별 `ReportFeedback`을 enqueue한다. AFTER_COMMIT 이후 기존
비동기 흐름으로 이전 코칭을 생성한다. 현재 ACTIVE 리포트는 코칭을 만들지 않는다.

사용자가 HTML에서 현재 최종 수확을 등록하면 기존 ACTIVE 리포트가 현재 COMPLETED
리포트로 전환된다. 이전 리포트는 이미 피드백 행이 있으므로 멱등성 검사에서 다시
enqueue되지 않고, 현재 작업별 리포트 코칭과 새 수확 기록 코칭만 시작된다.

서버가 막 시작된 직후 현재 최종 수확을 즉시 제출하면 이전 코칭이 아직 진행 중일 수
있어 호출이 일부 겹칠 수 있다. 이를 막기 위한 새 작업 큐나 동기 대기는 추가하지
않는다. 로컬 HTML은 이전 리포트 코칭 상태를 먼저 보여 주고, 이전 작업들이
`READY` 또는 `FAILED`가 된 뒤 현재 최종 수확을 제출하도록 안내한다.

### 직전 주기 비교의 정확한 범위

현재 `ReportFeedbackPreviousReport`는 직전 완료 리포트의 동일 작업 통계만 가진다.
이전 주기의 개별 기록과 메모는 현재 코칭 프롬프트에 넣지 않는다. 따라서 모델이
전년 메모의 `물이 고임`, `대응이 늦음` 같은 문장을 직접 인용하거나 현재 메모와
질적으로 비교하게 만들지 않는다.

이번 시드에서 실제 비교 가능한 값은 다음과 같다.

- 물 준 횟수, 간격, 양과 방법의 분포
- 거름 총량과 평균량
- 병 관리에 사용한 약과 물의 양
- 풀을 정리한 횟수와 방법
- 현재 최종 수확 등록 후 수확량과 기른 기간

현장 메모의 전년·현재 서사는 기록 목록의 현실감과 각 주기 자체 코칭에 사용한다.
이전 메모 비교가 제품 요구가 되면 `previousReport`에 동일 작업 기록을 추가하는 별도
설계로 다루며 이번 변경에 끼워 넣지 않는다.

### 과거 기록의 live weather 제외

`RecordFeedbackContextAssembler`는 현재 날짜와 대상 기록 날짜의 차이를 확인한다.
테스트에서 시간을 고정할 수 있도록 기존 서비스 패턴처럼 기본값이
`Clock.systemDefaultZone()`인 `Clock`을 주입한다.

```text
기록 날짜가 오늘부터 0~7일 전
  -> 기존처럼 현재 날씨와 7일 예보 조회

기록 날짜가 7일보다 오래됐거나 미래
  -> weather = null
  -> warning = weather_skipped_for_historical_record
  -> weatherPort 호출 안 함
```

7일은 현재 예보 조회 범위와 같은 경계다. 기록에 저장된 당시 날씨와 온도는 계속
컨텍스트에 포함한다. 따라서 2025년 10월 수확을 2026년에 등록해도 2026년 현재
날씨를 근거로 다음 행동을 만들지 않는다.

### local 서버 접근 경계

고정 로컬 계정은 public 로그인 API로 사용할 수 있으므로 CORS와 `.gitignore`만으로는
LAN 접근을 막을 수 없다. `application-local.yml`에 다음 기본값을 둔다.

```yaml
server:
  address: ${SERVER_ADDRESS:127.0.0.1}
```

이 설정은 local 프로필에만 적용한다. 물리 기기에서 개발 서버에 접속해야 할 때만
개발자가 `SERVER_ADDRESS=0.0.0.0`을 명시하고 같은 네트워크 노출을 인지한다. dev와
prod의 바인딩에는 영향을 주지 않는다.

## 기록·리포트 공통 쉬운 말 정책

### 적용 대상

쉬운 말 검증은 실제 API에서 사용자에게 노출되는 생성 문장에만 적용한다.

```text
RecordFeedback
- goodPoint.text
- nextActions[].text

ReportFeedback
- summary
- strengths[].text
- improvements[].text
- nextActions[].text
```

`basis`, `evidenceRefs`, 통계 맵, 입력 스냅샷, RAG 검색 질의, 공식 문서 본문은 내부
데이터이므로 이 정책으로 거부하지 않는다. Record API의 `due`, `category`와 Report
API의 `workType`도 구조용 enum 계약이므로 그대로 둔다.

### 공통 정책의 최소 책임

`coaching/common`에 `CoachingTextPolicy` 하나를 둔다. 이 객체는 다음 두 책임만 가진다.

1. 두 프롬프트가 공유하는 사용자 문장 작성 지침을 제공한다.
2. 생성된 사용자 문장에 영문자나 확인된 어려운 용어가 있는지 검사한다.

공통 생성 서비스나 validator 상속 계층은 만들지 않는다. 두 기존 validator가 정책의
결과를 각자의 안전한 warning 코드로 변환한다.

### 쉬운 표현 기준

초기 금지 목록은 실제 입력과 기존 테스트에서 확인된 표현으로 제한한다.

| 쓰지 않을 표현 | 사용자 문장 표현 |
| --- | --- |
| 관수 | 물 주기 |
| 시비 | 거름 주기 |
| 방제 | 병이나 벌레 관리 |
| 병해충 | 병이나 벌레 |
| 생육 | 자라는 상태 |
| 정식 | 모종 심기 |
| 파종 | 씨 뿌리기 |
| 제초 | 풀 뽑기 |
| 전정 | 가지 정리 |
| 과습 | 물기가 너무 많음 |
| 배수 | 물 빠짐 |
| 추대 | 꽃대가 올라옴 |
| 하엽 | 아래쪽 잎 |
| 토양 | 흙 |
| 수분 | 물기 |
| 살포 | 뿌리기 |
| 살균제 | 병을 막는 약 |
| 유기질 | 동식물 재료로 만듦 |

사용자 문장에 `[A-Za-z]`가 하나라도 있으면 거부한다. 따라서 `WATERING`, `DRIP`,
`recordCount`, `kg`, `pH` 모두 허용하지 않는다. 필요한 단위는 `킬로그램`,
`밀리리터`, `리터`처럼 한국어로 쓴다. JSON 필드명과 구조용 enum은 이 검사 대상이
아니다.

금지 용어는 단순하고 예측 가능한 literal substring으로 검사한다. 이 목록은 생성
코칭에서 농업 의미로 쓰이는 것이 확인된 표현만 둔다. `약해`처럼 `잎이 약해 보여요`
라는 일상 표현과 농약 피해 뜻이 겹치는 문자열은 validator 금지 목록에 넣지 않고,
프롬프트에서 농약 피해를 `약 때문에 생긴 피해`로 풀어 쓰도록 안내한다.

새 용어는 실제 사용자 노출 문제나 회귀 사례가 생겼을 때만 추가한다. 모든 농업
단어를 미리 사전에 넣거나 문장의 의미를 추측하는 validator는 만들지 않는다.

### 모델 입력 표현

두 PromptBuilder는 모델이 raw 코드를 복사하지 않도록 사용자 문맥을 쉬운 표현으로
조립한다.

- `WorkType.name`이나 도메인 `label` 대신 코칭 전용 쉬운 작업 이름을 사용한다.
- 작업 상세 enum은 raw name이나 기존 label을 그대로 쓰지 않고 명시적인 쉬운 표현으로
  바꾼다.
- `code`와 `label`이 함께 있는 상세·통계 값은 둘 다 그대로 출력하지 않고 code를
  코칭 전용 쉬운 표현으로 변환한다.
- `seedAmount`, `irrigationMethod` 같은 필드명 나열 대신 `씨앗 양`, `물을 준 방법`
  같은 문장으로 표현한다.
- 기록 프롬프트의 schema version, 검색 query reason, 날씨 source, raw risk flag처럼
  코칭 작성에 불필요한 내부 표시는 사용자 문맥에서 제거한다.
- 영농 경영 형태는 현재 코칭 판단에 사용되지 않으므로 prompt 표시에서 제거한다.
- 리포트 프롬프트의 대상 작업은 쉬운 작업 이름만 표시한다.
- 리포트의 상세·통계는 `Map.toString()`으로 출력하지 않는다. code/label 쌍, 단위와
  자주 사용하는 통계 필드를 한국어 문장으로 조립한다.
- 공식 기술 문서의 원문 용어는 근거 정확도를 위해 보존하되, 시스템 지침에서 그대로
  복사하지 말고 일상말로 풀도록 요구한다.

작업 이름은 다음과 같다.

| WorkType | 코칭·HTML 표시 |
| --- | --- |
| `PLANTING` | 심기 |
| `WATERING` | 물 주기 |
| `FERTILIZING` | 거름 주기 |
| `PEST_CONTROL` | 병이나 벌레 관리 |
| `WEEDING` | 풀 뽑기 |
| `PRUNING` | 가지 정리 |
| `HARVEST` | 수확 |
| `ETC` | 기타 작업 |

`PLANTING`은 씨앗과 모종·번식 재료를 모두 포함하므로 범용 이름을 `씨 뿌리기`로
고정하지 않는다. 상세 표현은 `PropagationMethod`에 따라 `씨앗을 뿌림`, `가지를
잘라 심음`, `접붙임`, `가지를 묻어 뿌리내림`, `포기를 나눠 심음`, `키운 모종을
심음`처럼 나눈다.

기존 enum label 자체에도 어려운 용어가 있으므로 다음과 같은 코칭 전용 매핑을
사용한다.

| 내부 값 예 | 코칭 입력 표현 |
| --- | --- |
| `DRIP` | 호스로 조금씩 물을 줌 |
| `SPRINKLER` | 물을 뿌리는 장치로 줌 |
| `MANUAL` | 손으로 물을 줌 |
| `SOIL` | 흙에 거름을 줌 |
| `FOLIAR` | 잎에 거름물을 뿌림 |
| `ORGANIC_FERTILIZER` | 유기물로 만든 거름 |
| `FUNGICIDE` | 병을 막는 약 |
| `INSECTICIDE` | 벌레를 막는 약 |
| `HAND` | 손으로 풀을 뽑음 |
| `MACHINE` | 기계로 풀을 정리함 |
| `MULCHING` | 덮개를 깔아 풀을 막음 |
| `ROOT_BARK` | 뿌리와 껍질 |
| `CULTIVATED` | 밭에서 기름 |

나머지 enum 값도 같은 방식으로 각 enum의 전체 entries를 빠짐없이 매핑하는 테스트를
둔다. 새 enum이 추가되어 매핑이 누락되면 테스트가 실패해야 하며 raw name이나 기존
label로 조용히 fallback하지 않는다.

도메인의 `WorkType.label`은 검색, API 카탈로그, 리포트 통계에서 계속 사용되므로
변경하지 않는다.

### 출력 검증과 재시도

각 기존 validator는 공백, 길이, 존댓말, 근거 범위, 중복 검사 후 사용자 문장에
공통 정책을 적용한다.

```text
영문자 발견       -> *_text_language
금지 용어 발견    -> *_text_language
```

내부 `basis`에 전문용어나 enum이 있어도 API에 노출되지 않으므로 허용한다. 검증
warning에는 생성 원문이나 사용자 데이터를 넣지 않고 고정 코드만 사용한다.

첫 출력이 언어 정책을 어기면 기존 두 번째 구조화 출력 요청으로 다시 생성한다.
ReportFeedback의 안전한 재시도 warning 목록과 RecordFeedback generation processor의
진단 warning 허용식에 새 고정 코드를 추가한다. 재시도 횟수는 기존 최대 두 번을
유지한다.

두 번 모두 실패하면 해당 코칭을 `STRUCTURED_OUTPUT_INVALID`로 종료한다. 어려운
표현을 후처리해 저장하거나 검증을 audit warning으로 낮추지 않는다. 한 작업의
ReportFeedback 실패는 다른 작업의 상태에 영향을 주지 않는다.

## HTML 표시

API 구조를 바꾸지 않고 `frontend/dev-rag-test.html`에서 코칭 카드의 enum 표시를
명시적으로 바꾼다.

```text
WorkType
PLANTING -> 심기
WATERING -> 물 주기
FERTILIZING -> 거름 주기
PEST_CONTROL -> 병이나 벌레 관리
WEEDING -> 풀 뽑기
PRUNING -> 가지 정리
HARVEST -> 수확
ETC -> 기타

Due
TODAY -> 오늘
THIS_WEEK -> 이번 주
NEXT_WEEK -> 다음 주
NEXT_CHECK -> 다음 확인 때
```

RecordFeedback category도 같은 방식으로 `날씨`, `병이나 벌레`, `물 주기`, `거름
주기`, `약 뿌리기`, `수확`, `밭 관리`, `일반`으로 표시한다. JSON 응답 원문을
디버깅하는 영역은 개발자용이므로 wire 값 유지가 가능하지만 농부 관점의 코칭 카드에는
raw enum을 직접 출력하지 않는다.

HTML의 고정 농장 이름과 최종 수확 폼 기본값은 새 시드 페르소나 및 2025년 수확
시나리오와 함께 변경한다. 이전 리포트 피드백 상태를 먼저 확인하라는 안내도 표시한다.

## 테스트 전략

구현은 테스트 우선으로 진행한다.

### 공통 언어 정책

- 쉬운 한국어 문장은 위반 없음
- `WATERING`, `DRIP`, `recordCount`가 있는 문장은 언어 위반
- `10kg`, `pH`처럼 영문 단위가 있는 문장은 언어 위반
- 금지 목록의 용어가 포함된 문장은 언어 위반
- 숫자, 날짜, 한국어 단위가 있는 문장은 허용
- WorkType과 작업 상세 enum의 모든 entries에 코칭 전용 쉬운 표현이 존재
- raw name이나 기존 도메인 label로 fallback하지 않음

### RecordFeedback

- PromptBuilder가 raw 상세 enum과 불필요한 내부 값을 사용자 문맥에 넣지 않음
- 기존 근거, 말투, 길이, 안전 지침은 유지
- goodPoint와 nextAction text의 영어·용어 위반을 validator가 거부
- 내부 basis의 enum과 전문용어는 허용
- 7일 이내 기록은 live weather를 조회
- 7일보다 오래된 기록과 미래 기록은 weather port를 호출하지 않고 고정 warning을 남김
- 첫 응답이 언어 위반이고 두 번째가 쉬운 문장이면 재시도 후 성공
- 두 번 모두 위반하면 `STRUCTURED_OUTPUT_INVALID`

### ReportFeedback

- PromptBuilder가 대상 작업을 raw `WorkType.name`으로 표시하지 않음
- 기록 상세에서 enum code를 그대로 표시하지 않음
- summary와 세 목록의 text를 모두 검사
- 빈 strengths/improvements/nextActions 허용 계약 유지
- 직전 비교는 동일 작업 통계만 사용하고 이전 기록 메모를 전달하지 않음
- 언어 위반 warning이 안전한 고정 코드로 재시도 프롬프트에 전달됨
- 작업 하나의 실패가 형제 작업에 전파되지 않는 기존 계약 유지

### 로컬 시드

- 고정된 이전 완료 주기와 현재 ACTIVE 주기를 생성
- 이전 `HarvestRecord.isFinalHarvest=true`가 정확히 하나 존재
- 현재 주기에는 최종 수확이 없음
- 이전 최종 수확 시각이 현재 첫 기록보다 빠름
- 두 주기에 비교 대상 작업 타입이 공통으로 존재
- 모든 메모가 비어 있지 않고 `로컬`, `테스트`, `샘플`을 포함하지 않음
- 작업별 상세 저장 수가 기록 수와 일치
- 중복 이메일이면 Harvest repository와 projection service를 포함한 후속 의존성을
  호출하지 않음
- 시드 후 projection rebuild가 한 번 호출됨

시드 테스트는 전체 메모 문장을 모두 고정하지 않는다. 주기 경계, 작업 타입, 핵심
금지어, 상세 저장 계약만 보호해 문체 조정을 불필요하게 막지 않는다.

### 검증 명령

모든 Gradle 명령은 wrapper가 있는 `backend`에서 실행한다.

```bash
cd backend
```

먼저 Git으로 추적되는 운영 코드의 관련 테스트를 실행한다.

```bash
./gradlew :application:test \
  --tests '*CoachingTextPolicyTest' \
  --tests '*RecordFeedbackPromptBuilderTest' \
  --tests '*RecordFeedbackContextAssemblerTest' \
  --tests '*RecordFeedbackOutputValidatorTest' \
  --tests '*RecordFeedbackGenerationServiceTest' \
  --tests '*ReportFeedbackPromptBuilderTest' \
  --tests '*ReportFeedbackOutputValidatorTest' \
  --tests '*ReportFeedbackGenerationServiceTest'

./gradlew :api:test \
  --tests '*RecordFeedbackControllerTest' \
  --tests '*ReportFeedbackControllerTest'
```

현재 로컬 workspace에만 존재하는 ignore 파일은 별도로 검증한다. 이 명령은 해당
파일이 없는 깨끗한 checkout이나 CI에서는 실행 대상이 아니다.

```bash
./gradlew :application:test --tests '*DevRagSeedServiceTest'
./gradlew :api:test --tests '*DevRagSeedRunnerTest'
```

그 뒤 전체 영향 모듈을 검증한다.

```bash
./gradlew --no-parallel :domain:test :application:test :api:test --rerun-tasks
```

마지막으로 로컬 DB를 재생성해 서버를 시작하고 다음을 수동 확인한다.

1. local 서버가 기본적으로 `127.0.0.1`에만 바인딩된다.
2. 이전 COMPLETED 리포트와 현재 ACTIVE 리포트가 보인다.
3. 이전 작업별 코칭이 서버 시작 후 생성된다.
4. HTML 코칭 카드에 raw enum이나 영어 메타데이터가 보이지 않는다.
5. 현재 최종 수확을 제출하면 현재 리포트와 수확 기록 코칭이 비동기로 생성된다.
6. 과거 수확 기록 코칭에 현재 날씨나 향후 예보가 들어가지 않는다.
7. 생성된 모든 사용자 문장이 친근한 존댓말을 쓰고 영어·확인된 금지 용어를 포함하지 않는다.
8. 현재 작업별 리포트 코칭이 직전 동일 작업 통계를 비교한다.

ignore된 로컬 시드·runner·HTML과 그 테스트는 현재 workspace의 수동 QA 자산이다.
저장소의 깨끗한 checkout과 CI에서 재현되는 완료 근거는 추적되는 공통 정책,
PromptBuilder, validator, generation 및 API 테스트로 한정해 보고한다.

## 오류 처리와 남는 위험

- 작물 시드가 없으면 기존처럼 로컬 시드 생성을 명확히 실패시킨다.
- 기존 시드 회원이 남아 있으면 멱등성 가드가 새 데이터를 적용하지 않는다. 로컬 DB
  재생성이 필요하다.
- 이전 리포트 선투영 직후 현재 최종 수확을 즉시 제출하면 비동기 호출이 일부 겹칠
  수 있다. 이번 범위에서는 전용 큐나 실행기 제한을 추가하지 않는다.
- 개발자가 `SERVER_ADDRESS=0.0.0.0`으로 local loopback 제한을 해제하면 고정 로컬
  계정이 같은 네트워크에서 접근 가능해진다. 이는 물리 기기 테스트에만 명시적으로
  사용한다.
- 엄격한 쉬운 말 검증 때문에 모델이 두 번 모두 금지 표현을 쓰면 해당 항목이
  `STRUCTURED_OUTPUT_INVALID`가 된다. 금지 표현을 사용자에게 노출하는 것보다 실패를
  명확히 보여 주는 쪽을 선택한다.
- 초기 금지 목록 밖의 어려운 용어는 통과할 수 있다. 실제 노출 사례가 확인될 때만
  작은 회귀 테스트와 함께 목록을 확장한다.
- literal substring 정책은 일부 문맥을 보수적으로 거부할 수 있다. 모호한 단어는
  금지 목록에 넣지 않고 prompt 안내만 사용한다.
- 이미 READY인 기존 로컬 코칭은 자동 재생성되지 않는다. 새 정책 검증에는 DB 재생성이
  필요하다.
- 로컬 시드와 HTML은 Git에 추적되지 않으므로 다른 개발자 환경에 자동 배포되지 않는다.
  이는 자격 증명과 개인 테스트 데이터를 커밋하지 않는 기존 저장소 정책을 따른다.

## 완료 기준

- 이전 완료 주기와 현재 수확 전 주기가 현실적인 순서와 현장 메모로 생성된다.
- 서버 시작 시 이전 완료 리포트가 투영되고 이전 작업별 코칭이 먼저 시작된다.
- HTML에서 현재 최종 수확을 등록하면 현재 리포트 및 수확 기록 코칭이 생성된다.
- 과거 수확 기록 코칭은 현재 날씨와 향후 예보를 조회하거나 근거로 사용하지 않는다.
- RecordFeedback과 ReportFeedback의 모든 사용자 노출 문장에서 영문자, raw enum,
  초기 금지 농업 용어가 검증으로 차단된다.
- 현재 ReportFeedback의 이전 주기 비교는 동일 작업 통계 범위로만 표현된다.
- API wire enum과 검색용 기술 용어는 기존 계약을 유지한다.
- HTML 코칭 카드는 enum 메타데이터를 한국어로 표시한다.
- local 프로필 서버는 별도 override가 없으면 loopback에만 바인딩된다.
- 관련 테스트와 전체 domain/application/api 테스트가 통과한다.
- 추적되는 운영 코드 변경은 Lore 프로토콜을 지킨 작은 커밋들로 남긴다.
