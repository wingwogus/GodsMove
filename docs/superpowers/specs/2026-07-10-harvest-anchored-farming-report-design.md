# 수확 종료 기준 영농 리포트 설계

> 상태: 대체됨
>
> 새 기준 문서:
> `2026-07-10-cycle-report-record-coaching-rag-redesign.md`
>
> 이 문서의 수확 종료 경계 아이디어는 유지하지만, 조회 시 실시간 계산,
> 지난해 비교, AI 비범위 결정은 새 기준 문서가 대체한다.

## 1. 배경

기존 리포트 초안은 최근 7일, 최근 30일, 전체 기간, 사용자 지정 기간을
기준으로 영농기록을 집계한다. 이 방식은 조회는 단순하지만 작물, 지역,
노지·시설 재배에 따라 달라지는 실제 농사 주기를 반영하지 못한다.

이번 리포트의 목적은 달력 기간의 작업량을 보여주는 것이 아니라, 농부가
완료한 한 번의 농사 주기를 지난해 같은 농사 주기와 비교할 수 있게 하는
것이다.

## 2. 결정 요약

- 리포트는 `마지막 수확`으로 종료된 한 주기를 기준으로 생성한다.
- 모든 수확 기록에서 사용자가 `일부 수확` 또는 `마지막 수확`을 선택한다.
- `일부 수확`은 현재 주기를 종료하지 않는다.
- `마지막 수확`은 현재 주기를 종료하고 완결된 리포트의 기준점이 된다.
- 심기 기록은 주기의 강제 시작점이 아니라 주기 안의 중요 기준점이다.
- 이전 완료 주기가 있으면 그 주기의 마지막 수확 다음 기록부터 집계한다.
- 이전 완료 주기가 없으면 같은 밭·작물의 최초 기록부터 집계한다.
- 리포트 통계는 조회할 때 영농기록으로 다시 계산하며 별도 결과물로 저장하지
  않는다.
- AI 요약과 코칭은 이번 리포트 MVP에 포함하지 않는다.

이 설계는 기존 `BR-REPORT-002`의 기간 선택 방식과 기존
`GET /api/v1/reports/farming` 초안을 대체한다.

## 3. 목표와 비목표

### 목표

- 심기 기록이 없어도 완료된 농사 주기 리포트를 제공한다.
- 심기 전 밑거름·제초 등 준비 작업을 현재 주기에서 누락하지 않는다.
- 같은 밭·같은 작물의 지난해 완료 주기와 작업별 통계를 비교한다.
- 관수, 시비, 방제, 수확 등 작업 유형별 상세 데이터를 한 리포트에서 본다.
- 상자, 단, kg처럼 환산할 수 없는 단위를 왜곡 없이 표시한다.

### 비목표

- 작물별 표준 재배 시기나 지역별 생육 달력을 추정하지 않는다.
- AI가 주기 경계나 마지막 수확 여부를 판단하지 않는다.
- 상자·단을 kg으로 임의 환산하지 않는다.
- 진행 중인 주기의 실시간 코칭 리포트는 제공하지 않는다.
- 같은 밭·같은 작물을 동시에 여러 작기로 겹쳐 재배하는 경우는 MVP에서
  자동 분리하지 않는다.

## 4. 핵심 용어

- **일부 수확**: 같은 농사 주기에서 이후 수확이 더 예정된 수확 기록
- **마지막 수확**: 사용자가 해당 농사 주기의 종료라고 확인한 수확 기록
- **완료 주기**: 마지막 수확으로 시작과 끝의 조회 경계가 확정된 기록 묶음
- **최초 주기**: 해당 밭·작물에 앞선 마지막 수확 기록이 없는 첫 완료 주기
- **심기 기준점**: 완료 주기 안에서 가장 이른 `PLANTING` 기록
- **기록 기준 주기**: 심기 기록 없이 최초 또는 이전 종료 이후의 기록으로
  구성된 완료 주기

## 5. 주기 경계 규칙

주기는 반드시 회원, 밭, 작물 조합별로 계산한다.

### 5.1 종료점

선택한 `마지막 수확` 기록을 현재 주기의 종료점으로 사용한다. 종료 기록은
현재 주기에 포함한다.

### 5.2 시작점

1. 선택한 종료점보다 앞선 `마지막 수확` 기록을 같은 회원·밭·작물에서
   찾는다.
2. 이전 종료점이 있으면 그 기록 다음의 첫 영농기록이 현재 주기의 시작점이다.
3. 이전 종료점이 없으면 같은 회원·밭·작물에서 확인 가능한 최초 영농기록이
   시작점이다.
4. 삭제된 기록은 모든 경계 및 통계 계산에서 제외한다.

경계 비교는 `workedAt`을 우선 사용한다. 동일 시각 기록의 안정적인 정렬을
위해 `createdAt`, `id`를 보조 순서로 사용한다.

### 5.3 심기 기록

주기 안에 심기 기록이 있으면 가장 이른 심기를 기준점으로 사용한다.

- 시작점부터 심기 직전까지: `재배 준비`
- 심기부터 마지막 수확까지: `심기 이후`

심기 기록이 여러 개면 가장 이른 심기를 화면의 대표 기준점으로 사용하되,
모든 심기 기록을 작업 통계에는 포함한다.

심기 기록이 없으면 단계를 나누지 않고 시작점부터 마지막 수확까지를
`기록 기준 주기`로 표시한다. 이 경우 심기 이전 작업이 앱에 기록되지 않았을
수 있다는 데이터 완전성 안내를 표시한다.

### 5.4 예시

심기가 있는 최초 주기:

```text
03.01 밑거름
03.05 제초
03.12 심기
04.01 관수
08.10 일부 수확
08.20 마지막 수확
```

- 전체 리포트 범위: 03.01~08.20
- 재배 준비: 03.01~03.11
- 심기 이후: 03.12~08.20

심기가 없는 최초 주기:

```text
04.01 관수
04.15 시비
08.10 일부 수확
08.20 마지막 수확
```

- 전체 리포트 범위: 04.01~08.20
- 시작 기준: `최초 기록`
- 완전성 상태: `심기 기록 없음`

## 6. 수확 입력 흐름

수확 기록 저장 시 프론트엔드는 다음 선택을 필수로 받는다.

- `일부 수확`: 기록만 저장하고 주기를 유지한다.
- `마지막 수확`: 기록을 저장하고 완료 주기 리포트를 생성할 수 있게 한다.

`마지막 수확` 저장이 성공하면 사용자는 바로 이번 농사 리포트로 이동할 수
있다. 리포트는 저장 시점의 스냅샷이 아니라 조회 시점의 최신 기록으로
계산한다.

마지막 수확 기록의 구분을 변경하거나 기록을 삭제하면 주기 경계가 달라진다.
이 작업 전에는 과거 리포트가 합쳐지거나 사라질 수 있음을 확인받아야 한다.

## 7. 저장 모델

리포트 결과 또는 별도 작기 엔티티는 MVP에서 저장하지 않는다. 기존
`HarvestRecord`에 다음 필드를 추가한다.

```text
isFinalHarvest: Boolean
```

- `false`: 일부 수확
- `true`: 마지막 수확

기존 수확 기록은 안전하게 `false`로 마이그레이션한다. 시스템이 과거 기록을
임의로 마지막 수확으로 지정하지 않는다. 기존 사용자에게 과거 주기 비교가
필요하면 과거 수확 기록을 직접 마지막 수확으로 지정해야 한다.

완료 주기의 식별자는 별도 ID를 만들지 않고 마지막 수확의
`FarmingRecord.id`를 사용한다. 마지막 수확 기록이 존재하는 동안 같은 주기를
안정적으로 다시 조회할 수 있다.

## 8. 리포트 조회와 비교 주기

### 8.1 현재 주기 조회

리포트 조회는 마지막 수확 기록 ID를 받는다. 서버는 소유권과
`isFinalHarvest = true`를 검증한 뒤 앞선 마지막 수확까지 역으로 조회하여
주기 범위를 계산한다.

### 8.2 지난해 주기 선택

비교 후보는 다음 조건을 모두 만족해야 한다.

- 같은 회원
- 같은 밭
- 같은 작물
- 완료 주기
- 종료 연도가 현재 주기 종료 연도의 직전 연도

후보가 하나면 자동 선택한다. 후보가 없으면 비교 영역을 숨기고
`비교할 지난해 완료 주기 없음` 상태를 표시한다. 후보가 둘 이상이면 시스템이
임의로 대응시키지 않고 사용자가 비교 주기를 선택한다. 이는 시설 재배나
연중 다회 재배에서 잘못된 주기를 비교하는 일을 막기 위함이다.

비교는 두 완료 주기의 전체 범위를 사용한다. 주기 길이가 다를 수 있으므로
작업 횟수와 총량뿐 아니라 주기 일수, 회당 평균, 평균 작업 간격을 함께
표시한다.

## 9. 화면 구성

리포트는 별도 종류로 나누지 않고 단일 상세 화면을 사용한다.

### 상단 요약

- 작물명과 밭 이름
- 현재 주기 시작일·종료일과 총일수
- 시작 기준: `최초 기록`, `이전 주기 종료 이후`, `심기 기록 있음`
- 비교 중인 지난해 주기
- 총 작업 횟수
- 수확량 단위별 합계

### 작업 유형별 탭 또는 펼침 섹션

- 파종/정식
- 관수
- 시비
- 병해충 방제
- 제초
- 전정
- 수확

기록이 없는 작업 유형은 `기록 없음` 상태를 보여주며 0을 성과처럼 강조하지
않는다.

### 기록 내역

현재 완료 주기에 포함된 원본 기록을 시간순으로 확인한다. 사용자는 잘못
포함된 기록을 수정할 수 있으며, 수정 후 리포트는 다시 계산된다.

## 10. 작업별 통계

### 공통

- 작업 횟수
- 작업한 날짜 수
- 평균 작업 간격
- 첫 작업일과 마지막 작업일
- 지난해 주기 대비 횟수·간격 변화

평균 간격은 작업 기록이 두 건 이상일 때만 계산한다.

### 파종/정식

- 종자량: 단위별 합계
- 모종 수: 단위별 합계
- 종자 출처 비율
- 구입처별 횟수

kg과 g은 명시적인 단위 변환 후 합산할 수 있다. 개수 단위는 서로 호환되는
경우에만 합산한다.

### 관수

- 관수 횟수와 평균 간격
- 관수량 단계별 비율: 적음, 보통, 충분
- 관수 방식별 비율

현재 구조에는 실제 물의 부피가 없으므로 총 사용 수량은 제공하지 않는다.

### 시비

- 시비 횟수와 평균 간격
- 자재별 사용량과 구성비
- 시비 방식별 횟수
- 회당 평균 사용량

### 병해충 방제

- 방제 횟수와 평균 간격
- 농약명별 사용량
- 총 살포량
- 방제 대상별 횟수

ml과 g은 서로 합산하지 않는다.

### 제초

- 제초 횟수와 평균 간격
- 제초 방식별 횟수와 비율

### 전정

- 전정 횟수와 평균 간격
- 사진 포함 기록 수와 비율

전정 상세 테이블이 없으므로 수량이나 방식 통계는 제공하지 않는다.

### 수확

- 일부·마지막 수확을 포함한 수확 횟수
- 수확량 단위별 합계
- 단위별 회당 평균 수확량
- 약용 부위별 수확 횟수
- 재배·채취 출처 비율
- 재배 기간 단위별 분포

kg, 단, 상자는 환산 기준이 없으므로 별도 합계로 표시한다. 지난해 비교도
같은 단위끼리만 계산한다. 지난해에 같은 단위가 없으면 증감률을 표시하지
않는다.

## 11. 증감 계산 규칙

- 횟수 증감: 현재 횟수 - 지난해 횟수
- 총량 증감률: `(현재 총량 - 지난해 총량) / 지난해 총량 * 100`
- 회당 평균: 총량 / 해당 단위 기록 수
- 평균 간격: 시간순으로 인접한 작업일 간격의 평균

지난해 값이 0이면 백분율을 계산하지 않고 절대 차이만 표시한다. 서로 다른
단위를 합산한 종합 증감률은 만들지 않는다. 화면의 지표는 계산 근거가
불명확한 종합 점수를 사용하지 않는다.

## 12. API 방향

기존 수확 기록 생성·수정 요청과 응답에 `isFinalHarvest`를 추가한다.

완료 주기 목록과 상세 리포트는 다음 책임으로 분리한다.

```text
GET /api/v1/farming-reports/cycles
GET /api/v1/farming-reports/cycles/{finalHarvestRecordId}
```

목록 조회는 밭과 작물 필터를 지원한다. 상세 응답은 현재 주기 통계,
지난해 비교 후보, 선택된 비교 주기 통계를 포함한다. 후보가 여러 개면
비교할 마지막 수확 기록 ID를 쿼리로 전달한다.

모든 조회는 인증된 회원 소유의 밭·작물·기록으로 제한한다.

## 13. 데이터 흐름

1. 사용자가 수확 기록과 마지막 수확 여부를 저장한다.
2. 마지막 수확이면 완료 주기 목록에 해당 기록이 나타난다.
3. 사용자가 리포트를 조회한다.
4. 서버가 직전 마지막 수확과 현재 마지막 수확 사이의 기록을 조회한다.
5. 서버가 심기 기준점과 데이터 완전성 상태를 계산한다.
6. 작업 유형별 집계기가 현재 주기 통계를 계산한다.
7. 지난해 비교 후보를 찾고, 비교 주기가 확정되면 같은 방식으로 집계한다.
8. 서버가 통계와 원본 기록 요약을 반환한다.

작업 유형별 집계기는 서로 독립적으로 두어 상세 필드가 없는 작업이나 새
작업 유형이 추가되어도 전체 리포트 계산에 영향을 최소화한다.

## 14. 예외 및 경계 상황

- 수확 기록만 있는 주기도 허용하되 시작과 종료가 같은 기록임을 표시한다.
- 심기 기록이 없으면 리포트 생성을 막지 않는다.
- 일부 수확은 여러 건 존재할 수 있으며 마지막 수확 전까지 같은 주기에
  포함한다.
- 마지막 수확을 잘못 지정하면 사용자가 기록을 수정해 경계를 바로잡는다.
- 과거 날짜로 기록을 추가하면 `workedAt` 기준으로 해당 완료 주기에 포함되고
  리포트가 다시 계산된다.
- 마지막 수확 삭제 또는 일부 수확으로 변경 시 해당 완료 주기는 더 이상
  독립된 리포트로 조회되지 않는다.
- 같은 밭·같은 작물의 주기가 서로 겹치는 재배는 현재 경계 규칙만으로
  분리할 수 없다. 이 요구가 생기면 별도 재배 묶음 ID를 도입한다.

## 15. 검증 전략

### 도메인 및 애플리케이션 테스트

- 일부 수확은 주기를 종료하지 않는다.
- 마지막 수확은 완료 주기를 만든다.
- 이전 마지막 수확은 제외하고 다음 기록부터 현재 주기에 포함한다.
- 최초 주기는 최초 기록부터 포함한다.
- 심기 전 밑거름·제초가 재배 준비에 포함된다.
- 심기가 없을 때 기록 기준 주기로 계산된다.
- 삭제된 기록은 경계와 통계에서 제외된다.
- 과거 기록 추가·수정 후 리포트가 다시 계산된다.
- 평균 간격과 0 분모 증감 규칙을 지킨다.
- kg, 단, 상자를 서로 합산하지 않는다.

### API 테스트

- 수확 생성·수정 시 마지막 수확 여부를 주고받는다.
- 다른 회원의 마지막 수확 ID로 리포트를 조회할 수 없다.
- 지난해 후보가 0개, 1개, 여러 개인 응답을 검증한다.
- 후보가 여러 개인 경우 명시적으로 선택한 주기만 비교한다.
- 마지막 수확이 아닌 기록 ID로 상세 리포트를 조회하면 거부한다.

### 프론트엔드 테스트

- 수확 입력에서 일부·마지막 수확을 선택한다.
- 마지막 수확 저장 후 완료 리포트로 이동할 수 있다.
- 심기 없음과 지난해 비교 없음 상태를 올바르게 표시한다.
- 복수 지난해 후보에서 사용자가 비교 주기를 선택할 수 있다.
- 수확량의 여러 단위가 각각 표시되고 레이아웃이 깨지지 않는다.

## 16. 성능 고려

주기 조회의 핵심 조건은 회원, 밭, 작물, 작업일, 삭제 여부다. 리포트 구현 시
실제 실행 계획을 확인하고 다음 복합 인덱스를 검토한다.

```text
(member_id, farm_id, crop_id, worked_at, is_deleted)
```

작업 유형별 상세 테이블은 주기 범위의 기록 ID를 먼저 제한한 뒤 조회한다.
목록과 상세를 한 번에 과도하게 조인하지 않는다.

## 17. 후속 확장

- 진행 중인 주기 미리보기
- 사용자가 명시적으로 만드는 재배 묶음
- 같은 밭·작물의 중첩 작기 지원
- 상자당·단당 실측 중량을 입력한 경우 kg 환산
- 완료 리포트에 연결되는 별도 AI 코칭

이 확장은 현재 MVP의 마지막 수확 경계와 작업별 통계 계약을 유지하면서
추가할 수 있어야 한다.
