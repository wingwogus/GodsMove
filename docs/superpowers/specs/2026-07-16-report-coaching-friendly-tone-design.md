# 리포트 코칭 친근한 문체와 행동 지향 개선점 설계

## 목적

리포트 코칭이 내부 보고서처럼 딱딱하게 들리거나, 개선점이 "정보가 없어 해석이
제한됐어요"라는 한계 설명으로 끝나는 문제를 줄인다. 농부에게 보이는 모든 리포트
코칭 문장을 쉬운 말과 부드러운 존댓말로 작성하고, 개선점에는 무엇을 보완해야 하는지
알 수 있는 방향을 반드시 담는다.

## 현재 문제

현재 `ReportFeedbackPromptBuilder`는 섹션별 개수, 20~65자 길이, 근거 참조,
존댓말을 지정한다. 하지만 `improvements`가 담당할 의미는 별도로 정의하지 않고,
`nextActions`에만 실행 방법이 드러나야 한다고 지시한다.

`ReportFeedbackOutputValidator`도 구조, 길이, 항목 수, 허용 근거만 검사한다. 따라서
다음과 같이 판단의 한계만 설명하는 문장도 정상 결과로 저장될 수 있다.

> 공식 지침의 파종은 면적 기준으로 판단한다는 점을 반영해 이번 심은 양은 면적
> 정보가 없어 해석이 제한됐어요.

비교 입력도 `직전 완료 리포트`, `직전보다`, `33.33퍼센트` 같은 표현을 사용해
모델이 시스템 용어와 불필요한 소수점을 사용자 문장에 옮길 가능성이 있다.

## 범위

- `ReportFeedbackPromptBuilder`의 리포트 코칭 섹션별 역할과 문체 지침
- 모델에게 전달하는 지난 재배 비교 표현
- 비교 변화율의 사용자용 정수 반올림
- `ReportFeedbackPromptBuilderTest`의 회귀 계약

## 제외 범위

- 개별 영농기록 코칭 문체
- API 응답과 구조화 출력 필드
- 데이터베이스 스키마와 기존 저장 결과
- `previousReport` 같은 내부 클래스·필드 이름
- 자연어 후처리와 문구 강제 치환
- 문체 전용 출력 검증기와 새 재시도 오류 코드
- 비교 계산기의 정밀도와 내부 통계값

## 변경 경계

변경은 `ReportFeedbackPromptBuilder`와 관련 단위 테스트로 한정한다. 기존
`ReportFeedbackContent`, `ReportFeedbackOutputValidator`, 생성 재시도 흐름은
수정하지 않는다.

서버가 계산한 정확한 비교값과 근거 참조는 그대로 유지한다. PromptBuilder가 모델에게
전달하는 문장만 농부가 읽기 좋은 표현으로 바꾼다. 따라서 API와 저장 계약에는 영향이
없다.

## 공통 문체 계약

`summary`, `comparisons`, `strengths`, `improvements`, `nextActions`의 사용자 노출
문장에 다음 원칙을 적용한다.

- 쉬운 일상말과 친근한 존댓말을 사용한다.
- 내부 보고서나 시스템을 설명하는 말투를 쓰지 않는다.
- `공식 지침`, `직전 리포트`, `해석이 제한됐어요` 같은 표현을 그대로 노출하지
  않는다.
- 농부가 남긴 행동이나 기록을 먼저 말하고, 판단과 안내를 자연스럽게 연결한다.
- 근거에 없는 사실, 권장 수치, 작업 결과를 만들지 않는다.
- 기존 20~65자 계약 안에서 핵심만 전달한다.
- 서로 다른 섹션에서 같은 내용을 그대로 반복하지 않는다.

공식 기술 문서를 근거로 사용하더라도 문서 자체를 문장의 주어로 내세우지 않는다.
문서의 판단 기준을 농부가 이해할 수 있는 기록 항목이나 작업 맥락으로 풀어 쓴다.

## 섹션별 역할

### 요약

이번 재배에서 확인한 핵심을 균형 있게 요약한다. 보고서를 평가하는 표현보다 실제
작업과 기록을 중심으로 쓴다.

> 이번 심기는 작업 내용은 잘 남겼지만 밭 넓이 기록이 더 필요했어요.

### 지난 재배와 비교

서버가 계산한 동일 작업의 변화 사실만 전달한다. 칭찬, 문제 진단, 권고는 넣지 않는다.
사용자 문장과 모델 입력에서 `직전` 대신 `지난 재배`를 사용한다.

> 지난 재배보다 심기 기록이 한 번 늘었어요.

### 잘한 점

근거에서 확인되는 구체적인 행동과 그 행동이 도움이 된 이유를 연결한다. 근거 없는
포괄적인 칭찬은 피한다.

> 심은 방법과 양을 함께 남겨 작업 내용을 확인하기 좋았어요.

### 개선점

다음 세 요소를 한 문단 안에서 자연스럽게 연결한다.

1. 이번 기록이나 작업에서 부족했던 점
2. 그 부족함이 판단이나 관리에 미치는 영향
3. 앞으로 보완할 방향

자료가 부족한 경우에도 "판단하기 어려웠어요" 또는 "해석이 제한됐어요"로 끝내지
않는다. 판단에 필요한 기록 항목을 알려 주고, 다음에는 무엇을 함께 남기면 좋은지
안내한다.

> 밭 넓이가 없어 심은 양이 알맞았는지 보기 어려웠어요. 다음에는 두 정보를 함께
> 기록해 보세요.

### 다음 행동

다음 작업에서 바로 실행할 수 있는 한 가지 행동을 구체적으로 안내한다. 개선점의
방향을 그대로 반복하지 않고, 언제 무엇을 기록하거나 확인할지 한 단계 더 명확하게
쓴다.

> 다음 심기 때 밭 넓이와 심은 양을 같은 기록에 남기세요.

## 비교값 표현

`ReportFeedbackComparisonCalculator`가 계산한 `relativeChangePct`의 정밀도는
유지한다. PromptBuilder가 모델 입력을 조립할 때만 변화율의 절댓값을
`RoundingMode.HALF_UP`으로 정수 반올림한다.

- `33.33` -> `33퍼센트`
- `33.50` -> `34퍼센트`
- `-12.50` -> 감소 방향을 유지한 `13퍼센트`
- `0.49` -> `0퍼센트`

증가와 감소 방향은 기존 `difference` 부호로 판단한다. 변화율을 계산할 수 없는
경우에는 `변화율은 계산하지 않았어요`라는 문장을 모델 입력에서 제거하고, 확인 가능한
절대 변화만 제공한다.

예를 들어 모델 입력은 다음처럼 구성한다.

> 지난 재배보다 기록 횟수가 1회 늘었고 변화율은 33퍼센트예요.

## 데이터 흐름

```text
ReportFeedbackComparisonCalculator
  -> 정밀한 비교값과 변화율 유지
ReportFeedbackPromptBuilder
  -> 지난 재배 표현 적용
  -> 변화율을 사용자용 정수로 반올림
  -> 섹션 역할과 문체 계약을 모델에 전달
ChatClient 구조화 응답
  -> 기존 정규화와 검증
  -> 기존 API·DB 계약으로 저장
```

근거 식별자인 `report:<id>`, `record:<id>`와 공식 문서 ID는 바꾸지 않는다.

## 실패와 재시도

구조화 응답 파싱 실패, 길이 위반, 필수 항목 수 위반, 근거 참조 오류는 기존 최대 두
번의 생성 시도와 안전한 오류 코드 흐름을 그대로 사용한다.

이번 변경은 의미 검증기를 추가하지 않는다. 문체가 기대보다 덜 자연스럽더라도 기존
구조·길이·근거 계약을 통과하면 저장된다. 초기 단계에서는 프롬프트 역할과 회귀 예시로
품질을 유도하고, 실제 생성 결과에서 반복되는 명확한 위반이 확인될 때만 별도 검증을
검토한다.

## 테스트 전략

`ReportFeedbackPromptBuilderTest`에 다음 회귀 계약을 추가하거나 갱신한다.

- 다섯 섹션의 역할과 공통 문체 지침이 시스템 프롬프트에 포함된다.
- 개선점은 부족한 점, 판단에 미치는 영향, 개선 방향을 요구한다.
- 자료가 부족하면 필요한 기록 항목을 안내하고 한계 설명으로 끝내지 않도록 지시한다.
- 모델 입력에서 `직전 완료 리포트`, `직전보다`가 사라지고 `지난 재배`를 사용한다.
- `33.33퍼센트`는 `33퍼센트`로 전달한다.
- 반올림 경계인 `33.50퍼센트`는 `34퍼센트`로 전달한다.
- 감소율은 절댓값을 반올림하면서 감소 방향을 유지한다.
- 변화율이 없으면 절대 변화만 전달하고 `계산하지 않았어요`를 넣지 않는다.
- 기존 길이, 항목 수, 근거 참조 관련 테스트가 계속 통과한다.

구현 후 다음 순서로 검증한다.

```bash
cd backend
./gradlew :application:test --tests '*ReportFeedbackPromptBuilderTest'
./gradlew :application:test --tests '*ReportFeedbackOutputValidatorTest' \
  --tests '*ReportFeedbackGenerationServiceTest'
./gradlew test
```

## 완료 기준

- 리포트 코칭의 모든 사용자 문장이 쉬운 말과 부드러운 직접형 문체를 따르도록
  프롬프트가 지시한다.
- 개선점이 한계 설명으로 끝나지 않고 보완 방향을 포함한다.
- 다음 행동은 개선 방향을 실행 가능한 한 가지 행동으로 구체화한다.
- 지난 재배 비교에서 `직전`이라는 시스템 용어와 소수점 변화율을 모델에 전달하지
  않는다.
- 비교 계산 정밀도, 구조화 응답, API, DB, 근거 검증에 회귀가 없다.
- 관련 집중 테스트와 전체 백엔드 테스트가 통과한다.

## 남은 위험

프롬프트만으로 자연어 품질을 완전히 강제할 수는 없다. 라이브 모델이 역할을 일부
어길 가능성은 남는다. 다만 의미를 추측하는 검증기를 조기에 추가해 정상 응답까지
거부하는 위험보다 변경 범위를 작게 유지하는 편을 우선한다.
