# 리포트 피드백 재시도 교정 지시 설계

## 목적

리포트 코칭이 `STRUCTURED_OUTPUT_INVALID`로 실패했을 때 모델에게 내부 진단 코드만
전달하는 현재 재시도 방식을 개선한다. 첫 응답이 길이, 항목 수, 근거 참조 또는 JSON
형식을 어겼다면 어느 필드를 어떻게 고쳐야 하는지 구체적인 한국어로 알려 한 번의
재시도를 더 효과적으로 사용한다.

## 현재 문제

첫 생성 프롬프트는 `summary`와 모든 섹션의 `text`를 20~65자로 작성하도록 명시하고,
서버도 같은 범위로 검증한다. 그러나 검증 실패 후 모델에게 전달하는 내용은 다음과
같은 내부 코드에 그친다.

```text
- improvement_text_length
```

이 코드에는 실제로 몇 자인지, 어느 방향으로 고쳐야 하는지, 해당 필드에서 어떤
의미를 유지해야 하는지가 없다. OpenClaw는 네이티브 `response_format`을 지원하지
않으므로 서버가 Spring AI의 JSON Schema 프롬프트와 검증 재시도를 이용해 출력 형식을
유도해야 한다.

## 목표

- 첫 생성과 서버 검증에 동일한 20~65자 계약을 사용한다.
- 길이 위반 재시도에는 필드 경로, 실제 글자 수, 허용 범위와 수정 방향을 전달한다.
- 다른 안전한 검증 코드도 모델이 바로 실행할 수 있는 한국어 지시로 변환한다.
- 모델 원문과 허용되지 않은 근거 식별자는 재시도 프롬프트와 로그에 노출하지 않는다.
- 최대 생성 시도는 기존과 같이 2회로 유지한다.

## 제외 범위

- 45~55자 같은 별도 목표 범위 도입
- 서버의 문자열 자동 자르기, 요약 또는 강제 치환
- `ReportFeedbackContent`와 API 응답 구조 변경
- 데이터베이스 스키마 또는 실패 코드 변경
- 생성 시도 횟수 증가
- OpenClaw가 지원하지 않는 네이티브 `response_format` 재도입

## 검토한 접근

### 선택: 기존 계약과 구체적인 재시도 지시

첫 생성은 기존 20~65자 계약을 유지한다. 검증 실패 시 서버가 안전한 진단 정보만
사용해 실행 가능한 교정 문장을 만들고 한 번 재생성한다. 저장 계약과 생성 계약이
일치하고, 문장을 기계적으로 자르지 않으면서 현재 재시도보다 성공 가능성을 높인다.

### 제외: 45~55자 완충 목표

65자 상한을 넘을 가능성을 줄일 수 있지만 모델에게 전달하는 목표와 서버의 실제 허용
범위가 달라진다. 사용자가 생성과 검증에 같은 계약을 적용하는 편을 선택했으므로
도입하지 않는다.

### 제외: 서버 자동 축약

저장 성공률은 높일 수 있지만 부족한 점, 영향, 보완 방향 중 일부가 유실되거나 문장이
어색해질 수 있다. 코칭 의미를 서버가 추측해서 다시 쓰지 않는다.

## 생성 프롬프트

`ReportFeedbackPromptBuilder`의 기존 섹션 역할과 20~65자 범위는 유지한다. 길이
계약에는 공백과 문장부호를 포함하며, JSON을 반환하기 전에 `summary`와 각 `text`가
범위 안인지 확인하도록 한 문장을 추가한다.

별도 목표 범위, 글자 수를 맞추기 위한 무의미한 표현, 문장 강제 자르기는 허용하지
않는다.

## 재시도 데이터 흐름

```text
첫 생성
  -> Spring AI JSON Schema 프롬프트로 응답 요청
  -> ReportFeedbackContent 변환
  -> ReportFeedbackOutputValidator 검증
  -> 실패 코드와 안전한 수치로 한국어 교정 지시 생성
  -> 교정 지시를 사용자 프롬프트 끝에 추가해 한 번 재생성
  -> 동일한 변환과 20~65자 검증
  -> 성공 시 저장, 실패 시 STRUCTURED_OUTPUT_INVALID
```

`ReportFeedbackGenerationService`는 기존 안전 진단 코드를 최종 실패 원인과 로그에
계속 사용한다. 모델에게 전달할 교정 문장만 별도로 만들며, 원문을 포함하지 않는다.

## 교정 지시 규칙

### 길이 위반

길이 위반은 해당 필드가 하나 존재할 때 실제 `String.length`를 계산한다. 65자를
넘으면 줄이라고 안내하고, 20자 미만이면 섹션 역할을 유지하며 늘리라고 안내한다.

```text
summary는 현재 71자입니다. 공백과 문장부호를 포함해 20~65자로 줄이세요.

improvements[0].text는 현재 18자입니다. 부족한 점, 영향, 보완 방향을 유지하면서
공백과 문장부호를 포함해 20~65자로 늘리세요.
```

섹션별 의미 보존 지시는 다음과 같다.

- `comparisons[0].text`: 지난 재배와 달라진 사실만 유지한다.
- `strengths[0].text`: 잘한 행동과 도움이 된 이유를 유지한다.
- `improvements[0].text`: 부족한 점, 영향, 보완 방향을 유지한다.
- `nextActions[0].text`: 언제 무엇을 할지 한 가지 행동을 유지한다.

항목 개수가 잘못되어 특정 인덱스를 안전하게 참조할 수 없으면 실제 길이를 추측하지
않고 먼저 해당 배열을 정확히 1개로 고치도록 안내한다.

### 구조와 근거 위반

- `*_count`: 해당 배열은 정확히 1개여야 한다고 안내한다.
- `comparison_not_available`: `comparisons`를 빈 배열로 만들도록 안내한다.
- `*_basis_blank`: `basis`에 판단 근거를 작성하도록 안내한다.
- `*_text_blank`: `text`에 해당 섹션 역할의 문장을 작성하도록 안내한다.
- `*_evidence_refs_blank`: 허용 목록의 식별자를 하나 이상 사용하도록 안내한다.
- `unknown_evidence`: 잘못된 값을 되풀이하지 않고 허용 목록에 있는 값만 사용하도록
  안내한다.
- 비교 리포트 참조 누락: 현재 및 지난 재배 리포트 근거를 모두 포함하도록 안내하되
  UUID를 교정 문장에 반복하지 않는다.
- `structured_output_parse_failed`: 설명이나 Markdown 없이 JSON Schema의 필드명과
  타입을 그대로 따른 완전한 JSON을 반환하도록 안내한다.
- 인식하지 못한 진단: 원문을 노출하지 않고 모든 JSON 필드와 검증 조건을 다시
  확인하라는 일반 지시로 대체한다.

여러 오류가 있으면 중복을 제거하고 모든 교정 문장을 한 번의 재시도 프롬프트에 함께
전달한다.

## 보안과 진단 경계

- 모델이 만든 전체 응답과 사용자 기록 문장은 오류 메시지에 포함하지 않는다.
- 허용되지 않은 `evidenceRefs` 실제 값은 모델에게 되돌려 보내거나 로그에 남기지
  않는다.
- 재시도에는 고정된 필드 경로, 실제 글자 수, 허용 범위와 안전한 수정 동작만 넣는다.
- 최종 `ReportFeedbackGenerationFailure`의 cause에는 기존 안전 진단 코드만 남긴다.
- DB에는 기존처럼 `STRUCTURED_OUTPUT_INVALID`만 저장하며 이번 범위에서 진단 컬럼을
  추가하지 않는다.

## 변경 경계

- `ReportFeedbackPromptBuilder`: 반환 전 글자 수 확인 지시 추가
- `ReportFeedbackGenerationService`: 안전 진단 코드를 구체적인 재시도 문장으로 변환
- 관련 프롬프트·생성 서비스 단위 테스트 갱신

새 의존성과 새 추상화는 추가하지 않는다. 재시도 문장 생성은 현재 재시도 흐름 가까이에
작은 전용 함수들로 두고, 검증기의 저장 계약은 변경하지 않는다.

## 테스트 전략

### 프롬프트 계약

- 첫 요청이 기존 20~65자 계약을 계속 사용한다.
- 반환 전에 공백과 문장부호를 포함한 글자 수를 확인하도록 지시한다.
- 45~55자 별도 목표가 포함되지 않는다.

### 재시도 지시

- 73자인 `improvements[0].text`는 필드 경로, 73자, 20~65자, 줄이기와 의미 보존
  지시를 받는다.
- 18자인 `improvements[0].text`는 필드 경로, 18자, 20~65자, 늘리기와 의미 보존
  지시를 받는다.
- 잘못된 항목 개수는 정확히 1개로 수정하라는 지시를 받는다.
- JSON 변환 실패는 완전한 JSON과 Schema 타입 준수 지시를 받는다.
- 알 수 없는 근거 식별자의 실제 값은 재시도 프롬프트에 포함되지 않는다.
- 재시도 후 유효한 응답은 저장 가능한 결과로 반환된다.
- 두 번째 응답도 잘못되면 기존 `STRUCTURED_OUTPUT_INVALID`와 안전 진단 코드로
  실패한다.

### 회귀 검증

```bash
cd backend
./gradlew :application:test --tests '*ReportFeedbackPromptBuilderTest' \
  --tests '*ReportFeedbackGenerationServiceTest' \
  --tests '*ReportFeedbackOutputValidatorTest'
./gradlew check --rerun-tasks
```

## 완료 기준

- 첫 생성과 서버 검증 모두 20~65자 계약을 사용한다.
- 길이 위반 재시도는 실제 필드와 글자 수를 알려 줄이거나 늘릴 방향을 명확히 전달한다.
- 구조와 근거 위반 재시도는 내부 코드 대신 실행 가능한 한국어 지시를 전달한다.
- 최대 2회 생성, API·DB·실패 코드와 근거 보안 계약에 회귀가 없다.
- 집중 테스트와 전체 백엔드 검사가 통과한다.

## 남은 위험

OpenClaw는 JSON Schema와 자연어 지시를 네이티브로 강제하지 않으므로 두 번째 응답도
계약을 어길 가능성은 남는다. 이번 변경은 자동 축약이나 무제한 재시도로 실패를 숨기지
않고, 한 번의 안전한 재시도가 모델에게 충분히 구체적인 정보를 주도록 개선하는 데
한정한다.
