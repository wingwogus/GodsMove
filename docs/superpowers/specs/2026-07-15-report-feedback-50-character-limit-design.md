# 리포트 피드백 50자 제한 설계

## 목적

리포트 피드백의 모든 사용자 노출 문구를 화면에서 짧고 일정하게 읽을 수 있도록
`summary`와 각 섹션의 `text`를 최대 50자로 제한한다.

## 범위

- `summary`: 최대 50자
- `comparisons[].text`: 최대 50자
- `strengths[].text`: 최대 50자
- `improvements[].text`: 최대 50자
- `nextActions[].text`: 최대 50자
- 내부 근거 필드인 `basis`와 `evidenceRefs`는 제한하지 않는다.
- 기록 피드백의 기존 15~60자 검증 범위는 변경하지 않는다.

## 동작

`ReportFeedbackPromptBuilder`는 `summary`와 모든 섹션의 `text`를 최대 50자로
작성하도록 명시한다. 길이를 맞출 때 문장을 자르지 않고 핵심 내용이 남도록 다시
쓰게 한다.

`ReportFeedbackOutputValidator`는 50자를 초과한 `summary`에
`summary_text_length`, 각 섹션의 `text`에 `<section>_text_length` 경고를 만든다.
50자는 허용하고 51자부터 거부한다. 기존 빈 값, 섹션 개수, 한 문단, 근거 참조
검증은 그대로 유지한다.

길이 경고는 `ReportFeedbackGenerationService`의 안전 재시도 목록과
`ReportFeedbackGenerationHandler`의 안전 진단 목록에 포함한다. 생성 결과가
50자를 초과하면 기존 구조화 출력 재시도 흐름으로 한 번 다시 생성하며, 원문은
재시도 프롬프트나 로그에 노출하지 않는다.

## 제외 사항

- 50자에서 문자열을 강제로 자르지 않는다.
- 최소 글자 수를 새로 추가하지 않는다.
- API 응답 구조와 데이터베이스 스키마를 변경하지 않는다.
- 이미 저장된 리포트 피드백을 일괄 수정하지 않는다.

## 검증

- 프롬프트가 `summary`와 모든 `text`의 최대 50자를 요구하는지 확인한다.
- `summary`와 섹션 `text`가 정확히 50자일 때 허용되는지 확인한다.
- 51자인 `summary`와 각 섹션 `text`가 안전한 길이 경고를 만드는지 확인한다.
- 길이 경고가 한 번의 재생성을 유도하고 원문을 노출하지 않는지 확인한다.
- 핸들러 로그에는 허용된 길이 진단 코드만 남는지 확인한다.
- 관련 집중 테스트와 전체 백엔드 테스트를 실행한다.

## 완료 기준

- 새로 생성되는 리포트 피드백의 `summary`와 모든 섹션 `text`가 50자 이하이다.
- 50자 초과 출력은 저장되지 않고 기존 재시도 흐름으로 처리된다.
- 기록 피드백과 리포트 피드백의 기존 구조·근거 검증 동작에 회귀가 없다.
