# 작업별 영농 리포트 대시보드 설계

## 목적

현재 로컬 리포트 콘솔의 원본 JSON 중심 화면을, 실제 앱에서 통계를 읽는 듯한
작업별 대시보드로 바꾼다. 전체 재배 주기 요약보다 선택한 작업 유형(관수·시비·방제·수확 등)의 기록 수, 간격, 분포, 누적량을 빠르게 읽는 것이 우선이다.

## 결정

기존 추적 파일인 `frontend/dev-rag-test.html`의 `리포트 통계` 탭을 확장한다.
별도 gitignore HTML 파일을 만들지 않는다. 이 파일은 이미 인증, 직접 farm/crop ID
입력, 현재·완료·상세 API 호출, 원본 응답 인스펙터를 갖고 있어 중복 콘솔보다
하나의 검증 도구가 낫다.

`test-ios`의 `Reports` 탭은 API 회귀 검증 목적을 유지한다. picker는 편의 기능으로
남기되, Farm ID와 Crop ID 텍스트 입력값을 조회 요청의 기준으로 사용한다. picker를
고르면 해당 ID가 입력칸에 채워지고, 사용자가 직접 UUID를 적으면 농장 목록을
불러오지 않아도 그 대상에 요청할 수 있다.

## 화면 구조

1. 상단에는 농장명·작물명·현재 주기 기간을 작은 맥락으로만 표시한다.
2. 작업 유형 탭은 `전체`, `파종`, `관수`, `시비`, `방제`, `제초`, `전정`, `수확`,
   `기타` 순서로 둔다. 탭은 해당 작업의 `recordCount`를 함께 표시한다.
3. 선택된 작업의 hero 요약은 작업명, 기록 수, 마지막 작업일, 평균 간격을 표시한다.
4. 공통 지표는 작업일 수, 기록률, 사진 첨부율, 평균 기온을 카드로 보인다.
5. 작업별 분포는 API가 제공하는 방식·양·자재·대상 분포를 막대 카드로 렌더링한다.
   수확은 총 수확량, 평균 수확량, amount coverage, 성장 기간을 전용 카드로 표시한다.
6. 완료 리포트 목록과 current/previous/selected 스냅샷 선택은 상단 컨트롤에 둔다.
   원본 응답, 요청 로그, API 오류는 기본 통계 화면에서 분리한 접힘 진단 영역에 둔다.

## 데이터 규칙

- `statistics`의 모든 작업 객체는 공통 지표를 직접 가진다. 없는 수치나 빈 분포는
  `-` 또는 "기록 없음"으로 표시하며 0으로 의미를 바꾸지 않는다.
- API가 아직 반환하지 않는 문구·점수·추천은 만들지 않는다.
- `전체` 탭은 여덟 작업의 recordCount 분포만 요약하고, 상세 분포는 작업 탭에서만 보인다.
- `current`, `previous`, `selected` 스냅샷 중 실제로 선택 가능한 값만 렌더링한다.

## 검증

- 기존 public report GET 세 개의 path/query/bearer 계약 XCTest는 유지한다.
- `test-ios` 전체 XCTest로 컴파일과 기존 하네스 회귀를 확인한다.
- `node --check frontend/dev-rag-test.html`로 인라인 JavaScript 문법을 확인한다.
- 브라우저에서 직접 Farm/Crop UUID 입력 → 현재 조회 → 관수 탭 → 완료 목록/상세
  조회의 golden path를 수동 확인한다. 실제 데이터가 없으면 API 호출·빈 상태·원본
  인스펙터까지 확인한다.
