# dev 기록 계약 병합과 모바일형 코칭 콘솔 설계

## 목적

`dev`의 최신 영농 기록 계약을 현재 `feat/coaching-rag` 브랜치로 병합하면서,
현재 브랜치에만 있는 기록 코칭과 작업별 리포트 코칭 동작을 보존한다. 병합된 최종
API 계약을 기준으로 로컬 HTML 콘솔을 첨부된 모바일 화면과 유사한 목록·상세 흐름으로
재구성한다.

HTML은 개발자가 로컬에서 실제 API를 확인하기 위한 도구다. 파일은 계속 Git에서
무시하며 커밋하지 않는다. 공유 가능한 결정과 검증 결과만 문서와 백엔드 커밋에
남긴다.

## 확정된 결정

- 병합 방향은 `dev`에서 `feat/coaching-rag`로 가져오는 방향이다.
- `dev`의 기록 입력·저장 계약을 기준으로 삼고 현재 브랜치의 코칭·리포트 생명주기를
  그 계약에 맞춘다.
- `frontend/dev-rag-test.html`은 로컬에서 수정하지만 `.gitignore`에서 제거하지 않고
  스테이징하거나 커밋하지 않는다.
- 기록 화면은 `목록 -> 상세 + 기록 코칭` 흐름으로 구성한다.
- 리포트 화면은 완료 주기를 선택한 뒤 실제 기록이 있는 작업을 세로 카드 목록으로
  보여 주고, 카드 선택 시 `작업 통계 + 작업 코칭 상세`로 이동한다.
- 기존 로그인, 토큰, 기록 CRUD, fixture, 상태 진단, 요청 로그는 접힌 `개발 도구`
  영역에 보존한다.
- 새 백엔드 합성 API, 범용 라우터, 프론트엔드 프레임워크, 공통 폴링 계층은 추가하지
  않는다.

## 접근 비교

### 선택: 의미 기반 병합과 단일 HTML 재구성

한 명의 병합 소유자가 추적되는 백엔드 파일의 충돌을 의미 단위로 해소한다. 동시에
다른 작업자는 Git에서 무시되는 HTML의 정적 모바일 레이아웃을 준비할 수 있다.
백엔드 계약이 테스트로 고정된 뒤 HTML의 요청 payload와 상세 필드 매핑을 최종
연결한다.

이 방식은 네트워크·인증·폴링 코드를 재사용하면서도 병합 중 같은 추적 파일을 여러
작업자가 수정하는 위험을 피한다.

### 기각: HTML을 먼저 완성한 뒤 dev를 병합

`dev`에서 심기, 시비, 방제, 수확 계약이 크게 바뀌었으므로 기존 계약으로 HTML을
완성하면 생성 payload와 상세 렌더링을 다시 작성해야 한다.

### 기각: 별도 HTML 또는 공통 JavaScript 모듈 분리

현재 콘솔은 로컬 단일 파일 도구다. 화면과 개발 도구를 별도 파일로 분리하면 인증,
상태, 폴링 공유를 위해 모듈 구조까지 추가해야 하며 현재 요구에는 과하다.

## dev 병합 경계

병합 기준은 로컬 `dev`와 `origin/dev`가 가리키는 `e4d1e31a`다. 병합 직전 두 참조가
같은지 다시 확인한다. 원격 갱신이 필요한 경우에는 먼저 fetch 결과를 확인하고 기준
커밋이 바뀌면 충돌 분석을 갱신한다.

### dev에서 채택할 기록 계약

- `PlantingMethod.SEED`와 `PlantingMethod.SEEDLING`으로 심기 입력을 나눈다.
- 씨앗 심기는 씨앗량·단위를, 모종 심기는 모종수·단위를 사용하며 번식 방법은 선택값이다.
- 거름량 단위는 `G`, `ML`을 사용하고 자재명을 저장한다.
- 병이나 벌레 관리 기록은 `pesticideId`와 선택적 `pestId` 참조를 사용한다.
- 수확 부위와 재배 기간은 선택값이며 주기 완료 기준은 `isLastHarvest`다.
- `EntryMode`는 문자열이 아니라 enum을 사용한다.
- 작업·관수·시비·제초 선택지의 최신 사용자용 라벨을 사용한다.

### 현재 브랜치에서 반드시 보존할 동작

- 회원 작물 검증
- 기록 `sourceRevision`
- 기록 변경에 따른 리포트 재투영
- 기록 저장 후 비동기 코칭 생성 요청
- 기록 변경·삭제 시 기존 코칭 stale 처리
- 기록 상세 교체 전 영속성 flush
- 작업별 리포트 코칭 준비·생성·조회 API와 독립 실패 처리

충돌 파일 전체를 `ours` 또는 `theirs`로 선택하지 않는다. DTO·Command·Result는 dev의
형태를 따르되 Service와 도메인에는 위 두 집합을 함께 남긴다.

### 자동 병합 뒤 별도 확인할 의미 충돌

텍스트 충돌이 없더라도 예전 `FertilizerMaterialCategory`, `PesticideCategory`,
`materialCategory` 검색 조건이 남으면 새 계약과 모순된다. 자재명과 농약 상품명·품목명
기준으로 정리한다.

`WorkTypeFieldCatalog`에는 dev에서 빠진 필수 Boolean `isLastHarvest` 필드를 추가해
실제 요청 DTO와 맞춘다.

거름 단위 `ML`을 기존 `amountKg` 또는 `totalAmountKg`에 넣거나 임의 환산하지 않는다.
통계가 단위를 안전하게 구분하지 못하는 범위에서는 비교와 합계를 생략한다. 수확의
선택 필드는 값이 없으면 통계·검색 질의·코칭 프롬프트에서도 해당 표현을 생략한다.

## HTML 화면 구조

`frontend/dev-rag-test.html`의 기존 API 요청, 인증, 폴링 함수를 재사용하고 DOM과
화면 상태만 재구성한다. 모바일 본문은 약 430~520px 폭으로 가운데 정렬하고 개발
도구를 펼쳤을 때만 넓은 진단 영역을 사용한다.

```text
영농 기록
├─ 기록
│  ├─ 기록 목록
│  └─ 기록 상세 + 참참참의 코칭
├─ 리포트
│  ├─ 완료 주기 선택
│  ├─ 작업별 카드 목록
│  └─ 작업 통계 + 작업 코칭 상세
└─ 개발 도구 (기본 접힘)
   ├─ 연결·인증·공통 ID
   ├─ 기록 생성·수정·삭제·fixture
   ├─ 리포트 수동 조회
   └─ 상태·요청·원본 응답 진단
```

### 기록 목록과 상세

목록은 `GET /api/v1/farming-records`를 사용한다. 작물, 작업, 시작일, 종료일 필터와
커서 기반 더 보기를 제공한다. 카드에는 작물명, 쉬운 작업명, 작업일, 날씨, 메모
미리보기, 실제 썸네일만 표시한다.

목록 응답에는 코칭 상태가 없으므로 각 카드마다 피드백을 조회하지 않는다. 항목을
선택했을 때만 아래 두 요청을 병렬로 호출한다.

```text
GET /api/v1/farming-records/{recordId}
GET /api/v1/farming-records/{recordId}/feedback
```

상세에는 작업명, 날짜·날씨, 메모, 작업별 상세 정보, 실제 작업 사진을 표시한다.
사진이 없으면 가짜 이미지 칸 대신 빈 상태 문구를 보여 준다. 코칭은 잘한 점과 다음
행동을 표시하고 `PENDING`이면 기존 2초 폴링을 이어 간다.

### 작업별 리포트 목록과 상세

완료 주기 목록에서 하나를 선택하면 리포트 상세와 작업별 코칭 목록을 각각 한 번
조회한다.

```text
GET /api/v1/farming-reports/{reportId}
GET /api/v1/farming-reports/{reportId}/feedback
```

통계의 `recordCount`가 0보다 큰 작업만 카드로 표시한다. 카드에는 쉬운 작업명,
기록 건수, 최근 작업일, 평균 간격, 코칭 상태와 준비된 요약 한 줄을 표시한다. 전체
작업 카드는 만들지 않는다.

작업 카드를 선택하면 해당 작업의 통계, 요약, 잘한 점, 개선점, 다음 행동을 표시한다.
이미 받은 피드백을 `workType`으로 찾아 렌더링하므로 탭이나 상세 전환마다 API를 다시
호출하지 않는다.

## 상태와 오류 처리

- 기록 상세를 닫거나 다른 기록으로 이동하면 이전 기록 피드백 폴링을 중지한다.
- 리포트 또는 완료 주기를 바꾸면 이전 리포트 피드백 폴링을 중지한다.
- 느린 이전 응답은 현재 선택 ID와 run ID가 다르면 렌더링하지 않는다.
- `PENDING`은 생성 중 문구, `READY`는 결과, `FAILED`는 안전한 failure code,
  `STALE`은 새 생성 필요 안내를 표시한다.
- 리포트 작업 하나의 실패는 다른 READY 작업 카드와 상세 표시를 막지 않는다.
- 인증·HTTP·파싱 오류는 현재 화면 안에 재시도 가능한 오류 상태로 표시하고 원본은
  개발 도구에서만 확인한다.
- API enum 이름은 사용자 화면에 직접 표시하지 않고 명시적 한국어 라벨을 사용한다.

## 병렬 작업 규칙

- 추적되는 병합·백엔드 파일은 한 명의 구현 소유자만 수정한다.
- HTML 작업자는 Git에서 무시되는 `frontend/dev-rag-test.html`만 수정한다.
- HTML의 정적 레이아웃은 병합과 병렬로 진행할 수 있지만 요청 payload와 상세 필드
  연결은 병합된 API 계약이 고정된 뒤 완료한다.
- 다른 작업자의 추적 파일 변경을 되돌리거나 전체 파일을 덮어쓰지 않는다.

## 검증

### 백엔드

- 병합 전 현재 브랜치 기준 테스트
- 충돌 해소 후 기록 controller/application/domain 집중 테스트
- 기록 코칭과 작업별 리포트 코칭 집중 테스트
- `:domain:test :application:test :api:test` 전체 회귀 테스트
- 이전 분류 enum과 잘못된 kg 환산 참조 정적 검색

### 로컬 HTML

- HTML 중복 ID와 인라인 JavaScript 문법 검사
- 기록 목록 필터·더 보기·상세 뒤로가기 확인
- 기록 상세와 코칭 병렬 조회, `PENDING/READY/FAILED/STALE` 확인
- 완료 주기 선택, 작업 카드 목록, 작업 상세 전환 확인
- 리포트 작업별 혼합 `PENDING/READY/FAILED` 확인
- 폴링 취소와 느린 응답 경쟁 조건 확인
- 첨부 화면과 같은 모바일 폭에서 가로 스크롤·잘림 여부 시각 검증
- `git status`와 `git check-ignore`로 HTML이 스테이징되지 않았음을 확인
