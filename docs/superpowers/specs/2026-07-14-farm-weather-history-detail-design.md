# 농장 날씨 기능 확장 — 기록 화면 과거 날짜 조회 + 날씨 상세 화면 보강

## Context

`ChamChamCham` 백엔드에는 이미 농장 위치 기준 "현재" 날씨를 조회하는 기능이 있다
(`FarmWeatherController` → `FarmWeatherService` → `WeatherProvider`/`KmaWeatherProvider`,
기상청 공공데이터포털 `VilageFcstInfoService_2.0`의 초단기실황/예보 사용).

두 가지 문제로 확장이 필요하다.

1. **기록하기 화면**: 농사 기록을 남길 때 선택한 날짜(과거 또는 당일)와 등록된 농지
   기준으로 그 날의 날씨상태(맑음 등)와 최저/최고기온을 보여줘야 하는데, 현재 구현은
   호출 시점("지금")만 지원한다. 이 값은 "그날 예보됐던 값"이 아니라 "그날 실제로
   있었던 값"(ASOS 종관기상관측 실측치)이어야 한다 — 기록은 사실을 남기는 것이
   목적이기 때문이다.
2. **날씨 상세 화면**: 항상 "당일" 기준으로 주소, 자외선지수, 강수확률, 습도, 풍속,
   그리고 오늘 포함 5일 간의 기온/날씨상태 패널을 보여줘야 한다. 이건 과거 날짜 문제와
   무관하고, 예보 API가 원래 하는 일(지금부터 미래)을 그대로 쓰면 되는 순수 확장이다.

두 기능은 서로 독립적이며, 신규/기존 코드 모두 `com.chamchamcham.*.weather` 패키지와
`FarmWeatherController`/`FarmWeatherService`, 그리고 `Farm`/`FarmRepository`(농지
도메인, 기록 도메인과 다름)에만 위치한다 — **`farming`(기록) 도메인 파일은 일절
건드리지 않는다.** 다른 세션이 기록(FarmingRecord) 도메인을 동시에 수정 중이므로
이는 병합 충돌을 원천적으로 피하기 위한 제약이며, 기존에도 기록 생성/수정은 날씨
서비스와 전혀 연결되어 있지 않아(클라이언트가 날씨를 조회해 값만 실어 보내는 구조)
이 제약은 기능 요구사항과도 자연스럽게 맞는다.

작업은 `dev` 기준의 새 브랜치 `feat/farm-weather-history-and-detail`에서 진행한다
(현재 `feat/search-enhancement`는 검색 기능 전용 브랜치라 여기서 진행하지 않는다).

## 검증 방법론

기상청/공공데이터포털 API 3종(ASOS 일자료, 단기예보 getVilageFcst, 생활기상지수
UV)에 대해 각각 독립된 조사 에이전트를 띄워 공식 문서(data.go.kr 상세페이지,
apihub.kma.go.kr 활용가이드 원문, 최소 1개 이상의 독립 블로그/코드 예제)를
교차 검증했다. 그 결과 최초 가정 중 다수가 실제로 틀렸거나 확인 불가능한 것으로
드러나 아래 설계에 전부 반영했다. **UV지수는 이 교차검증으로도 스펙이 확인되지
않아, "실제 서비스키로 먼저 확인" 스파이크 단계를 필수 선행 작업으로 포함시켰다**
— 확인 안 된 필드명을 사실인 것처럼 코드에 박아넣지 않기 위함이다.

## 조사로 확정된 API 선택

| 용도 | API | 비고 |
|---|---|---|
| 기록 화면 과거 날짜 날씨(실측) | 공공데이터포털 `AsosDalyInfoService/getWthrDataList` (지상 종관 ASOS 일자료) | `stnIds`(관측소 지점번호) 필요 — 위경도 아님. 새 관측소 매핑 컴포넌트 필요. 요청 파라미터는 가정대로 확인됨 |
| 날씨 상세 5일 패널 + 강수확률 | 기존 연동된 `VilageFcstInfoService_2.0`의 `getVilageFcst`(단기예보) | 기존 nx/ny 격자 그대로 재사용, 새 API 상품 아님. 단, 발표시각(base_time)에 따라 실제 제공 일수가 4~5일로 달라짐 |
| 날씨 상세 습도/풍속 | 이미 호출 중인 `getUltraSrtNcst` 응답에서 파싱만 추가 | 신규 외부 호출 없음 |
| 날씨 상세 주소 | `Farm.roadAddress` | 신규 외부 호출 없음 |
| 날씨 상세 자외선지수 | `LivingWthrIdxServiceV4/getUVIdxV4` (가칭) | 요청 파라미터/응답 필드명/PNU→법정동코드 매핑 타당성이 문서로 확인 안 됨 — 실서비스키 스파이크 필수 |

**기각한 대안**: 기록 화면 과거 날짜 조회를 `getVilageFcst`의 과거 `base_date` 재조회로
구현하는 방법도 검토했으나, 이는 "그날 실제 있었던 날씨"가 아니라 "그날 발표됐던
예보값"을 저장하게 되어 기록의 사실성을 해친다는 이유로 기각(사용자 확인 완료).

**운영 전제조건**: `AsosDalyInfoService`와 `LivingWthrIdxServiceV4`는 기존에 승인된
`VilageFcstInfoService_2.0`과 별개의 공공데이터포털 상품이라, 각각 별도 활용신청(승인)이
필요할 수 있다. `KMA_SERVICE_KEY` 값 자체는 계정 단위라 승인만 되면 재사용 가능.

## 핵심 설계 결정 (요약)

- 기록 화면 과거 날짜 조회는 ASOS 실측치 사용 (예보 재조회 방식은 기각).
- 날씨 상세 화면은 항상 "당일" 기준이라 과거 날짜 복잡성과 무관 — 순수 확장.
- 농지 미지정(`farmId` 없음) 호출 지원 — 현재는 첫 등록 농지로 기본값 처리하되,
  이 판단 로직을 `FarmWeatherService.resolveDefaultFarm` 한 곳에만 몰아넣어
  나중에 "마지막 선택 농지"/"기본 농지 지정" 기능으로 교체 가능하게 열어둔다.
- `getVilageFcst`의 5일 패널은 base_time에 따라 4~5일치만 존재할 수 있고, 오늘의
  TMN/TMX도 자주 없다는 것을 정상 상태로 받아들이고 TMP 폴백으로 보완한다.
- ASOS 어댑터는 JSON 단일값/배열 quirk와 `resultCode=03`(NODATA_ERROR) 모두 "데이터
  없음"으로 방어적으로 처리한다.
- UV지수는 스파이크 결과에 따라 스코프 포함 여부를 결정하며, 다른 모든 기능은 UV와
  무관하게 독립적으로 완결된다.

세부 구현 계획은 `docs/superpowers/plans/2026-07-14-farm-weather-history-detail-plan.md` 참조.

## 추가 결정 — 주간예보 "무조건 5일" 채우기 (2026-07-15)

`getVilageFcst`의 5일 패널이 발표시각(02/05/08/11/14시)에 따라 4일만 줄 때가 있다는 걸
정상 상태로 받아들이기로 했던 위 결정을, 이후 요구사항 변경으로 "항상 5일"로 바꿨다.
부족한 5번째 날("오늘+4일", 그글피)은 기상청 중기예보(`MidFcstInfoService`의
`getMidLandFcst`+`getMidTa`)로 백필한다. 실제 서비스키로 스파이크 검증한 결과:

- 육상예보(`getMidLandFcst`)와 기온(`getMidTa`)의 지역코드(regId) 체계가 서로 다르다
  (육상=도 단위, 기온=시/군 단위). 하지만 `getFcstZoneCd` 응답에서 시/군 세부지점에
  실제 위경도가 있고, 그 지점의 `regUp`을 2단계 거슬러 올라가면 정확한 육상예보용
  지역코드가 나온다는 것을 179개 지점 전수 검증으로 확인했다. 그래서 ASOS와 동일한
  Haversine 최근접 방식(`MidFcstStations`/`NearestMidFcstStationResolver`)으로 주소
  문자열 매칭 없이 좌표만으로 두 지역코드를 한 번에 구한다.
- "오늘+4일" 필드명(`wf4Am`/`wf4Pm`, `taMin4`/`taMax4`)을 실제 호출로 확정했다.
- 중기예보 `wf` 값은 "흐리고 비" 같은 자유 문장이라 단기예보와 같은 어휘(맑음/구름많음/
  흐림/비/눈)로 키워드 정규화한다.
- 이 백필은 UV와 같은 성격(있으면 좋고 없어도 전체는 안 죽는 보강)이라 `runCatching`으로
  감싸 실패 시 조용히 4일로 degrade한다 — "무조건 5일"은 정상 운영 조건에서의 보장이며,
  중기예보 자체가 완전히 장애 상태일 때만 예외적으로 4일이 될 수 있다.

세부 내용은 `~/.claude/plans/5-shiny-crystal.md` 참조.
