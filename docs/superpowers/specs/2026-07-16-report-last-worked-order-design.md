# 리포트 마지막 작업일 정렬 설계

## 배경

작업 리포트 목록은 현재 `ACTIVE` 상태를 먼저 묶고, 같은 상태 안에서는 재배 리포트의 종료일 또는 시작일을 기준으로 정렬한다. 같은 재배 리포트 안에서는 `WorkType` 선언 순서를 사용하므로 `PLANTING`이 실제 작업 시점과 관계없이 앞에 노출될 수 있다.

완료 리포트의 코칭 목록도 `WorkType` 선언 순서만 사용한다. 두 목록 모두 사용자가 최근에 수행한 작업을 먼저 찾으려는 흐름과 맞지 않는다.

## 목표

- 작업 리포트 목록 전체를 실제 마지막 작업일이 최신인 순서로 노출한다.
- 완료 리포트의 코칭 목록도 작업 종류별 마지막 작업일이 최신인 순서로 노출한다.
- 커서 페이지네이션의 정렬 키를 목록 정렬 키와 일치시켜 중복과 누락을 막는다.
- 같은 날짜와 비정상 통계에서도 결과 순서가 결정적이도록 한다.

## 비목표

- API 응답 필드를 추가하거나 제거하지 않는다.
- 리포트 생성 시점이나 통계 계산 방식을 변경하지 않는다.
- 데이터베이스 쿼리의 조회 범위나 현재 인메모리 작업 항목 전개 방식을 변경하지 않는다.
- 과거에 발급한 작업 리포트 목록 커서를 새 정렬 기준으로 변환하지 않는다.

## 선택한 정렬 규칙

### 작업 리포트 목록

다음 키를 왼쪽부터 순서대로 비교한다.

1. `lastWorkedOn` 내림차순. 값이 없는 항목은 마지막에 둔다.
2. 같은 작업일이면 `ACTIVE`, `COMPLETED`, `SUPERSEDED` 순서로 둔다.
3. 같은 상태이면 재배 리포트의 `sortAt`(`endsAt ?: startsAt`) 내림차순으로 둔다.
4. 같은 리포트 시각이면 `reportId` 내림차순으로 둔다.
5. 같은 리포트이면 `workType.ordinal` 오름차순으로 둔다.

이 규칙은 최근에 끝난 완료 재배의 작업이 오래된 진행 중 재배의 작업보다 앞에 올 수 있게 한다. 상태 우선 정렬은 마지막 작업일이 같은 경우에만 동률 처리에 사용한다.

### 완료 리포트 코칭 목록

한 리포트 안에서 다음 키를 비교한다.

1. 해당 `workType` 통계의 `lastWorkedOn` 내림차순. 값이 없으면 마지막에 둔다.
2. 같은 작업일이면 `workType.ordinal` 오름차순으로 둔다.

## 도메인 통계 접근

`CycleReportStatistics`에 `lastWorkedOnFor(workType: WorkType): LocalDate?`를 추가한다. 작업 리포트 투영과 코칭 목록 정렬이 동일한 통계 필드 매핑을 사용하게 해 작업 종류가 추가되거나 변경될 때 매핑이 어긋나는 위험을 줄인다.

기존 `recordCountFor`와 같은 명시적 `when` 매핑을 사용하며 별도 추상화나 의존성은 추가하지 않는다.

## 커서 설계와 호환성

작업 리포트 목록 커서에 `lastWorkedOn`을 필수 필드로 포함한다. 도메인 `WorkItemCursor`와 애플리케이션 `FarmingWorkReportCursorPayload`가 목록 비교에 필요한 다음 키를 모두 가진다.

- `lastWorkedOn`
- `status`
- `sortAt`
- `reportId`
- `workType`

커서 이후 항목 판정은 목록 comparator와 완전히 같은 키 순서를 사용한다. `lastWorkedOn`이 null이면 정렬상 가장 오래된 값으로 취급한다.

배포 전에 발급된 커서는 `lastWorkedOn` 필드가 없어 디코딩 시 `INVALID_CURSOR`가 된다. 커서는 불투명한 단기 페이지 토큰이므로 클라이언트가 첫 페이지부터 새로 조회하는 방식으로 복구한다. 서로 다른 정렬 의미를 가진 구형 커서를 억지로 수용해 중복 또는 누락을 만드는 것보다 명시적으로 거절하는 편을 선택한다.

## 오류 및 예외 데이터 처리

통계의 `recordCount`가 양수라면 일반적으로 `lastWorkedOn`도 존재한다. 다만 기존 데이터나 부분 마이그레이션을 고려해 nullable 계약은 유지한다.

- 값이 있는 작업은 모든 null 작업보다 앞에 둔다.
- null 작업끼리는 상태, 리포트 시각, ID, 작업 종류로 안정적으로 정렬한다.
- API 응답의 `lastWorkedOn` nullable 계약은 바꾸지 않는다.

## 검증 전략

- 도메인 저장소 테스트에서 오래된 ACTIVE 작업보다 최근 COMPLETED 작업이 먼저 오는지 검증한다.
- 같은 작업일에는 ACTIVE가 먼저 오고 이후 동률 키가 결정적 순서를 만드는지 검증한다.
- null 작업일은 목록 마지막에 오는지 검증한다.
- 커서 경계 전후로 항목의 중복과 누락이 없는지 검증한다.
- 애플리케이션 서비스 테스트에서 새 커서가 `lastWorkedOn`을 인코딩·디코딩하는지 검증한다.
- 코칭 조회 서비스 테스트에서 `WorkType` 선언 순서가 아니라 마지막 작업일 최신순으로 반환되는지 검증한다.
- 관련 모듈 테스트 후 backend 전체 `check --rerun-tasks`를 실행한다.
