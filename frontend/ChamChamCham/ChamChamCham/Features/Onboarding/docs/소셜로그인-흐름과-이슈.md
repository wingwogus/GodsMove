# 소셜 로그인 흐름 정리 (카카오/네이버/Apple)

> 개인 이해용 메모. AI 컨텍스트로 참조되는 문서가 아님 (AGENTS.md의 Product Source of Truth 목록에 포함 안 함).
> 코드가 바뀌면 이 문서는 낡을 수 있음 — 실제 동작 기준은 항상 코드.

## 1. 로그인 전체 흐름 (버튼 탭 → 토큰 저장 → 홈 화면)

### 클라이언트

1. **로그인 버튼** — `LandingView`에서 "카카오/네이버/Apple로 시작하기" 버튼을 누르면
   `AuthViewModel.loginWithKakao / loginWithNaver / loginWithApple` 호출.
   (`Features/Onboarding/Presentation/Views/LandingView.swift`)

2. **Provider SDK 브릿지 호출** — `AuthViewModel`이 각 provider의 SDK를 감싼 브릿지를 호출해서
   provider의 토큰을 받아옴.
   (`Features/Auth/Presentation/ViewModels/AuthViewModel.swift`)

   | Provider | 브릿지 | 반환값 | 비고 |
   |---|---|---|---|
   | 카카오 | `KakaoLoginBridge.login(nonce:)` | `idToken`(OIDC JWT) + `kakaoAccessToken` | 카카오톡 앱 전환 or 웹뷰(`ASWebAuthenticationSession`) |
   | 네이버 | `NaverLoginBridge.login()` | `accessToken` | OIDC 아님 — 순수 OAuth, idToken 없음 |
   | Apple | `AppleSignInCoordinator.signIn(hashedNonce:)` | `identityToken`(OIDC JWT) | 네이티브 `ASAuthorizationController` 사용 |

   이때 클라이언트가 `nonce`(재생 공격 방지용 랜덤 문자열)를 생성해서, **원본(raw) nonce는 백엔드로**,
   Apple의 경우 **해시된 nonce는 SDK 요청에만** 사용한다 (백엔드가 직접 해시해서 JWT의 nonce claim과 비교하기 때문).

3. **백엔드에 토큰 전달** — `AuthRepository`가 `AuthEndpoint.kakaoLogin / naverLogin / appleLogin`을 통해
   `POST /api/v1/auth/{provider}/login`으로 idToken/accessToken을 전달.
   이 시점엔 아직 우리 서비스 토큰이 없으므로 `requiresAuth: false`.
   (`Features/Auth/Data/AuthRepository.swift`, `AuthEndpoint.swift`)

4. **토큰 저장** — 백엔드가 응답한 `LoginResponseDTO`(accessToken/refreshToken + member + onboarding 상태)를
   받으면 `AuthRepository.persistingTokens`가 즉시 `AuthTokenStore.save`로 **Keychain**에 저장.
   (`Core/Networking/AuthTokenStore.swift`)

5. **로컬 상태 반영** — `AuthViewModel.handleLoginSuccess`가:
   - `MemberProfileCache`(SwiftData)에 member 정보 캐싱
   - `AppState.isAuthenticated = true`
   - `AppState.isOnboarded`를 서버가 알려준 값으로 세팅

6. **온보딩 이어가기** — `LandingView.login(with:)`이 `isOnboarded == false`면
   `OnboardingViewModel.jump(to: .basicProfile)`로 온보딩 단계 진입
   (프로필 입력 → 작물 선택 → 농지 위치 → 완료).
   마지막 `OnboardingViewModel.submit(appState:)`이
   `POST /api/v1/auth/onboarding/complete`를 호출해서 `member` + `farm` + `member_crop`을
   한 번에 생성하고 `appState.isOnboarded = true`로 변경.
   (`Features/Onboarding/Presentation/ViewModels/OnboardingViewModel.swift`)

7. **홈 화면 전환** — `RootView`가 `isAuthenticated && isOnboarded`가 둘 다 참이 되는 순간
   `AuthFlowView` 대신 `MainTabView`(홈)를 렌더링.
   앱 재실행 시엔 `RootView.bootstrap()`이 Keychain의 refreshToken + 로컬 SwiftData 캐시로
   오프라인 우선 자동 로그인을 시도한 뒤, 백그라운드에서 서버와 재조회(reconcile)함.
   (`App/RootView.swift`)

### 백엔드 (OIDC 검증, 요약)

`AuthController.kakaoLogin / appleLogin`이 각각 `KakaoLoginService` / `AppleLoginService`로 위임하고,
내부의 `KakaoOidcTokenVerifier.verify(idToken, expectedNonce)`가 다음을 검증한다:

- Kakao/Apple의 공개 JWK(discovery endpoint)로 idToken(JWT) **서명 검증**
- `issuer` / `audience`(우리 앱 native app key) / `exp` / `iat` **검증**
- 우리가 보낸 nonce와 JWT 안의 `nonce` claim이 **일치하는지 검증**

이 검증을 통과해야 "진짜 그 provider가 발급했고, 우리 앱을 대상으로 하며, 재사용되지 않은 토큰"이라고
신뢰하고 `member`를 조회/생성한 뒤 우리 서비스 자체의 accessToken/refreshToken을 발급해서 응답한다.

네이버는 OIDC가 아니라서 idToken 검증이 아니라, 받은 accessToken으로 네이버 프로필 API를
서버가 직접 호출해서 신원을 확인하는 방식이다.

## 2. 우리가 마주한 문제

독립된 버그 두 개가 겹쳐 있었다.

### (1) 크래시 — 카카오 로그인 취소 시 `SWIFT TASK CONTINUATION MISUSE`

- iOS `ASWebAuthenticationSession`(카카오톡 미설치 시 웹 로그인에 쓰임)의 completion handler가
  **취소 상황에서 두 번 호출되는 레이스 컨디션**이 있었음. Kakao SDK는 이 콜백을 그대로
  우리 코드에 전달만 하고 중복 호출을 막지 않음.
- `KakaoLoginBridge.login`은 이 콜백을 `withCheckedThrowingContinuation`으로 감싸는데,
  `CheckedContinuation`은 **정확히 한 번만** resume되어야 하는 계약이 있어서
  두 번째 호출 시점에 fatal error로 크래시.

### (2) 로그인 자체가 안 됨 (크래시를 고친 후 발견된 더 근본적인 문제)

- `Info-Additions.plist`의 `CFBundleURLTypes`에 등록된 **카카오 URL scheme이 플레이스홀더
  `kakaoREPLACE_ME` 그대로 방치**되어 있었음. `Secrets.swift`엔 실제 네이티브 앱 키가
  채워져 있었지만 plist는 업데이트가 안 된 상태였음.
- **네이버도 동일한 문제** — plist엔 `chamchamcham`으로 등록돼 있었지만, 실제
  `NaverLoginBridge`의 `NidOAuth.initialize(urlScheme:)`엔 `chamchamchamauth`로 되어 있어서
  서로 어긋나 있었음.
- 결과: 로그인 완료 후 provider(카카오톡 앱 / 네이버 앱 / 웹뷰)가 우리 앱으로 돌아오려고
  `kakao{키}://` 또는 `chamchamchamauth://`를 열어도, iOS가 "이 scheme을 가진 앱 없음"으로
  판단해서 조용히 실패 → **밖으로는 나가지는데 다시는 앱으로 안 돌아옴.**

즉 (1)만 고쳐도 (2) 때문에 여전히 로그인은 안 됐고, (2)까지 고치고 나서야 콜백이 실제로
앱으로 돌아와서 로그인이 완성됐다.

## 3. 해결 방법

1. **`KakaoLoginBridge.login`에 `didResume` 가드 추가**
   completion 클로저 안에서 이미 resume했는지 로컬 `var` 플래그로 추적해서, 두 번째 호출은
   무시하도록 처리. `ASWebAuthenticationSession`의 콜백은 항상 메인 스레드에서 오고
   `login`이 이미 `@MainActor`라서, 락(lock)이나 `nonisolated(unsafe)` 없이 평범한 `var`로
   충분함 (프로젝트 컨벤션상 concurrency escape hatch를 피하고 싶었음).

2. **`NaverLoginBridge.login`에도 동일한 가드 적용**
   네이버 SDK 소스(`WebAuthenticationService.swift`)를 확인해보니 네이버 앱 미설치 시
   인앱 브라우저 폴백에서 동일하게 `ASWebAuthenticationSession`을 쓰고 있어서, 같은
   레이스 컨디션 위험을 그대로 갖고 있었음. 아직 실제로 크래시가 나진 않았지만
   (레이스 컨디션이라 매번 재현되진 않음) 선제적으로 방어.

   Apple(`ASAuthorizationController`)은 손대지 않음 — 델리게이트 메서드가 정확히 한 번만
   불린다는 게 Apple의 API 계약이고, 코드도 이미 continuation을 옵셔널 프로퍼티로 두고
   두 델리게이트 콜백 모두 `defer { continuation = nil }`로 방어하고 있어서 안전함.

3. **`Info-Additions.plist`의 URL scheme 수정**
   - 카카오: `kakaoREPLACE_ME` → `kakao` + `Secrets.kakaoNativeAppKey` 값
   - 네이버: `chamchamcham` → `chamchamchamauth` (`NaverLoginBridge`가 실제 초기화에 쓰는 값과 일치)

   > 참고: `Info-Additions.plist`는 git에 커밋되는 파일이라(`Secrets.swift`와 달리 gitignore 대상 아님),
   > 카카오 네이티브 앱 키를 여기 직접 박아 넣게 됐다. Kakao 네이티브 앱키는 원래 앱 바이너리에
   > 그대로 노출되는 값이라 REST API 시크릿만큼 극비는 아니지만, 실제 키 값을 git에 안 올리는
   > 이 프로젝트의 기존 컨벤션과는 다르다는 점은 인지하고 있을 것.

두 수정 모두 커밋 완료 (`fix(onboarding): 카카오/네이버 로그인 콜백 크래시 및 미동작 수정`).

## 추가 메모 (2026-07-06 폴더 구조 리팩터)

이 문서 작성 이후, `AuthRepository`/`AuthEndpoint`/`KakaoLoginBridge`/`NaverLoginBridge`/
`AppleSignInCoordinator`/`AuthViewModel`을 `Features/Onboarding`에서 `Features/Auth`로 분리했다
(로그인은 온보딩 1회성 흐름이 아니라 로그아웃 후 재로그인 등 앱 생명주기 내내 재사용되는 세션
개념이라서). 위 경로들은 그 이전 기준이라 실제로는 `Features/Auth/Data`,
`Features/Auth/Presentation/ViewModels` 등으로 옮겨졌다 — 자세한 배경은 별도 아키텍처 문서
(`docs/superpowers/specs/2026-07-02-frontend-architecture-design.md`) 참고.
